SPA (React)

Uma Single Page Application roda inteiramente no browser — não pode guardar secrets. A solução é usar um public client com Authorization Code + PKCE.

⚛️

O desafio do SPA

Código no browser é visível para qualquer um. Não existe client_secret seguro. PKCE resolve isso sem precisar de secret.

Setup com react-oidc-context

A biblioteca react-oidc-context é um wrapper React sobre oidc-client-ts:

import { AuthProvider } from 'react-oidc-context';

const oidcConfig = {
  authority: 'https://keycloak/realms/meu-realm',
  client_id: 'minha-spa',
  redirect_uri: 'http://localhost:3000/callback',
  post_logout_redirect_uri: 'http://localhost:3000/',
  scope: 'openid profile email',
  // response_type defaults to 'code' (Authorization Code + PKCE)
};

function Root() {
  return (
    <AuthProvider {...oidcConfig}>
      <App />
    </AuthProvider>
  );
}

Usando Auth nos Componentes

import { useAuth } from 'react-oidc-context';

function Profile() {
  const auth = useAuth();

  if (auth.isLoading) return <div>Carregando...</div>;

  if (!auth.isAuthenticated) {
    return <button onClick={() => auth.signinRedirect()}>Login</button>;
  }

  return (
    <div>
      <p>Olá, {auth.user.profile.name}</p>
      <p>Email: {auth.user.profile.email}</p>
      <button onClick={() => auth.signoutRedirect()}>Logout</button>
    </div>
  );
}

Chamando APIs Protegidas

function Dashboard() {
  const auth = useAuth();
  const [data, setData] = useState(null);

  useEffect(() => {
    if (auth.isAuthenticated) {
      fetch('/api/data', {
        headers: {
          Authorization: `Bearer ${auth.user.access_token}`
        }
      })
        .then(res => res.json())
        .then(setData);
    }
  }, [auth.isAuthenticated]);

  return <div>{JSON.stringify(data)}</div>;
}

Silent Refresh

Tokens expiram. O silent refresh renova automaticamente sem redirecionar o usuário:

const oidcConfig = {
  // ...config anterior
  automaticSilentRenew: true,
  // Opção 1: via iframe (usa session cookie do Keycloak)
  silent_redirect_uri: 'http://localhost:3000/silent-renew.html',
  // Opção 2: via refresh token (se habilitado no client)
};

Segurança no Browser

✅ Boas práticas

Token em memória (variável JS)
PKCE obrigatório
Validar state parameter
CSP headers contra XSS
Considerar BFF para apps sensíveis

❌ Evitar

Token em localStorage (acessível via XSS)
Token em sessionStorage (mesmo risco)
Implicit flow (deprecated, inseguro)
Ignorar expiração do token
Client secret no código frontend

Padrão BFF (Backend for Frontend)

Para aplicações de alta segurança, o BFF mantém tokens no backend e usa cookies httpOnly com o frontend:

┌─────────┐     cookie      ┌─────────┐     token      ┌──────────┐
│  React  │ ──────────────→ │   BFF   │ ──────────────→ │   API    │
│  (SPA)  │ ←────────────── │ (proxy) │ ←────────────── │ (Resource│
└─────────┘   httpOnly       └─────────┘   Bearer JWT    │  Server) │
                                                          └──────────┘

O SPA nunca vê o token — elimina risco de XSS roubar credenciais.

Angular (alternativa)

Mesmos conceitos, biblioteca diferente: angular-auth-oidc-client. Configuração similar, usa módulos Angular em vez de hooks React.

🎯 Key Takeaway

SPA = public client + Authorization Code + PKCE. Tokens em memória, nunca em localStorage. Para apps de alta segurança, considere o padrão BFF.

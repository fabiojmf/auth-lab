Protegendo APIs (Resource Server)

Um Resource Server é qualquer API que recebe um token de acesso, valida-o e decide se concede ou nega acesso ao recurso solicitado.

🛡️

Duas formas de validar tokens

Validação local (JWT + assinatura) = rápido, sem rede. Introspection (pergunta ao Keycloak) = mais lento, mas verifica revogação.

Validação Local vs Introspection

⚡ Validação Local (JWT)

Verifica assinatura com chave pública (JWKS)
Sem chamada de rede na validação
Stateless — escala horizontalmente
Não detecta revogação em tempo real
✅ Use na maioria dos casos

🔍 Token Introspection

Envia token ao Keycloak para validação
Chamada de rede a cada request
Detecta revogação imediatamente
Funciona com opaque tokens
Use quando revogação em tempo real é crítica

Passos da Validação Local

Passo O que verificar Se falhar
1 Extrair token do header Authorization: Bearer xxx 401 — No token
2 Buscar JWKS do Keycloak (com cache!) 503 — Cannot validate
3 Verificar assinatura usando chave pública (matching kid) 401 — Invalid signature
4 Verificar exp (não expirado) 401 — Token expired
5 Verificar iss (issuer correto) 401 — Wrong issuer
6 Verificar aud (audience = esta API) 401 — Wrong audience
7 Extrair roles/permissions dos claims 403 — Forbidden (se role insuficiente)

Token Introspection

POST /realms/meu-realm/protocol/openid-connect/token/introspect
Content-Type: application/x-www-form-urlencoded

token=eyJhbGciOiJSUzI1NiIs...
&client_id=minha-api
&client_secret=secret-da-api

Resposta:

{
  "active": true,
  "sub": "f1234-abcd-5678",
  "email": "user@example.com",
  "realm_access": {
    "roles": ["user", "admin"]
  },
  "client_id": "app-frontend",
  "exp": 1716000000
}

Se "active": false, o token foi revogado ou expirou — negue acesso.

Configurando Audience

Por padrão, o Keycloak não inclui audience no access token. Sem audience, qualquer token do mesmo realm é aceito — inseguro.

Configuração no Keycloak:
1. Client Scopes → Criar scope "minha-api-scope"
2. Dentro do scope → Mappers → Add mapper
3. Mapper type: Audience
4. Included Client Audience: "minha-api"
5. Atribuir o scope ao client que precisa acessar a API

Roles vs Scopes

👤 Roles (coarse-grained)

Claim: realm_access.roles
Ou: resource_access.{client}.roles
Exemplo: admin, user, manager
Bom para: "quem pode acessar"

🔑 Scopes (fine-grained)

Claim: scope (space-separated)
Exemplo: read write delete
Delegado pelo usuário ao client
Bom para: "o que pode fazer"

Erros Comuns

Erro Causa Solução
Qualquer token do realm funciona Não valida audience Configurar Audience mapper + validar aud
Token válido rejeitado após key rotation JWKS cacheado para sempre Use createRemoteJWKSet do jose (já faz refresh automático ao encontrar kid desconhecido). Em frameworks manuais: cache com TTL (ex: 5 min) + refresh on unknown kid.
500 em vez de 401 quando token expira Exception não tratada Catch específico → retornar 401 com mensagem clara
Roles não reconhecidas Lendo claim errado Verificar se roles estão em realm_access ou resource_access

🎯 Key Takeaway

Sempre valide: assinatura + exp + iss + aud. Use validação local para performance e introspection quando precisa de revogação em tempo real. Nunca esqueça de configurar audience.

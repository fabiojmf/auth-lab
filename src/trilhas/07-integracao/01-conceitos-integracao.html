Conceitos de Integração

Antes de escrever código, você precisa entender como sua aplicação se relaciona com o Keycloak. Existem dois papéis fundamentais — e uma aplicação pode exercer ambos ao mesmo tempo.

🔌

Dois papéis de integração

Sua aplicação pode ser um Client (faz login de usuários) ou um Resource Server (protege APIs) — ou ambos.

Client (OAuth2 Client)

Um Client é qualquer aplicação que redireciona o usuário ao Keycloak para autenticação e recebe tokens de volta.

📱 O que faz

Redireciona usuário para login no Keycloak
Recebe authorization code no callback
Troca code por tokens (access + id + refresh)
Mantém sessão do usuário

📋 Exemplos

App web com login (Spring Boot MVC)
SPA React com botão de login
App mobile que abre browser para login
Backend que usa Client Credentials (M2M)

Resource Server

Um Resource Server é qualquer API que recebe tokens de um client, valida-os e decide se concede ou nega acesso.

🛡️ O que faz

Recebe token no header Authorization: Bearer xxx
Valida assinatura, expiração, issuer, audience
Extrai roles/permissions do token
Concede ou nega acesso ao recurso

📋 Exemplos

API REST Spring Boot
API Node.js/Express
Microserviço que só aceita chamadas autenticadas
API Gateway que valida tokens

Uma app pode ser ambos

Um backend web que faz login de usuários (Client) E expõe endpoints de API protegidos (Resource Server) exerce os dois papéis. Exemplo: um app Spring Boot com páginas web + endpoints REST.

Adapters vs Bibliotecas Padrão

❌ Modo antigo (deprecated)

keycloak-spring-boot-adapter
keycloak-js (adapter JavaScript)
keycloak-nodejs-connect
Acoplamento total ao Keycloak
⚠️ Deprecated desde Keycloak 19

✅ Modo novo (padrão OIDC)

spring-security-oauth2-*
oidc-client-ts / react-oidc-context
openid-client / jose
Funciona com qualquer provider OIDC
✅ Recomendado pelo próprio Keycloak

Por que a mudança? Adapters proprietários criam vendor lock-in. Com bibliotecas padrão, seu código funciona com Keycloak, Auth0, Okta, Azure AD — qualquer provider OIDC compatível.

Bibliotecas por Ecossistema

Ecossistema Biblioteca Uso
Java/Spring spring-security-oauth2-resource-server Proteger APIs (Resource Server)
Java/Spring spring-security-oauth2-client Login de usuários (Client)
Node.js jose Validação de JWT (leve, sem dependências)
Node.js openid-client (v6+) Client OIDC completo (login flows)
React oidc-client-ts / react-oidc-context SPA com PKCE
Angular angular-auth-oidc-client SPA Angular com PKCE
Python authlib / python-jose Client e validação de tokens

OIDC Discovery Endpoint

Todas as bibliotecas padrão usam o endpoint de discovery para se auto-configurar. Basta informar a URL do issuer — a biblioteca descobre sozinha os endpoints de token, JWKS, userinfo, etc.

GET https://keycloak/realms/meu-realm/.well-known/openid-configuration

Resposta (resumida):
{
  "issuer": "https://keycloak/realms/meu-realm",
  "authorization_endpoint": "https://keycloak/realms/meu-realm/protocol/openid-connect/auth",
  "token_endpoint": "https://keycloak/realms/meu-realm/protocol/openid-connect/token",
  "jwks_uri": "https://keycloak/realms/meu-realm/protocol/openid-connect/certs",
  "userinfo_endpoint": "https://keycloak/realms/meu-realm/protocol/openid-connect/userinfo",
  "end_session_endpoint": "https://keycloak/realms/meu-realm/protocol/openid-connect/logout"
}

Na prática, você configura apenas o issuer-uri e a biblioteca faz o resto.

🎯 Key Takeaway

Use bibliotecas OIDC padrão, não adapters do Keycloak. Seu código deve funcionar com qualquer provider OIDC. Configure apenas o issuer-uri — o discovery faz o resto.

Spring Boot

Spring Boot integra com Keycloak usando Spring Security OAuth2 — as mesmas bibliotecas que funcionam com qualquer provider OIDC. Nenhum adapter proprietário necessário.

🍃

Dois papéis no Spring Boot

OAuth2 Client = login de usuários via redirect. Resource Server = proteger endpoints de API validando JWT.

Dependências Maven

<!-- Para proteger APIs (Resource Server) -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-oauth2-resource-server</artifactId>
</dependency>

<!-- Para login de usuários (OAuth2 Client) -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-oauth2-client</artifactId>
</dependency>

Configuração: Resource Server

Para uma API que apenas valida tokens (não faz login):

# application.yml
spring:
  security:
    oauth2:
      resourceserver:
        jwt:
          issuer-uri: https://keycloak/realms/meu-realm

O Spring busca automaticamente o JWKS via discovery e valida assinatura, expiração e issuer.

Configuração: OAuth2 Client

Para uma aplicação web que faz login de usuários:

# application.yml
spring:
  security:
    oauth2:
      client:
        registration:
          keycloak:
            client-id: minha-app
            client-secret: meu-secret
            scope: openid,profile,email
        provider:
          keycloak:
            issuer-uri: https://keycloak/realms/meu-realm

Security Configuration

@Bean
SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http.authorizeHttpRequests(auth -> auth
        .requestMatchers("/public/**").permitAll()
        .requestMatchers("/admin/**").hasRole("admin")
        .anyRequest().authenticated()
    )
    .oauth2ResourceServer(oauth2 -> oauth2
        .jwt(jwt -> jwt.jwtAuthenticationConverter(jwtAuthenticationConverter()))
    );
    return http.build();
}

Extraindo Roles do Keycloak

O Keycloak coloca roles em claims não-padrão. O Spring não os reconhece automaticamente — você precisa de um converter:

📍 Onde o Keycloak coloca roles

realm_access.roles — roles do realm
resource_access.{client}.roles — roles do client

📍 Onde o Spring espera

scope claim (prefixo SCOPE_)
Ou authorities customizadas
@Bean
JwtAuthenticationConverter jwtAuthenticationConverter() {
    JwtGrantedAuthoritiesConverter grantedAuthorities = new JwtGrantedAuthoritiesConverter();

    JwtAuthenticationConverter converter = new JwtAuthenticationConverter();
    converter.setJwtGrantedAuthoritiesConverter(jwt -> {
        // Authorities padrão (scopes)
        Collection<GrantedAuthority> authorities =
            new ArrayList<>(grantedAuthorities.convert(jwt));

        // Extrair realm roles do Keycloak
        Map<String, Object> realmAccess = jwt.getClaimAsMap("realm_access");
        if (realmAccess != null) {
            List<String> roles = (List<String>) realmAccess.get("roles");
            if (roles != null) {
                roles.forEach(role ->
                    authorities.add(new SimpleGrantedAuthority("ROLE_" + role))
                );
            }
        }
        return authorities;
    });
    return converter;
}

Com esse converter, hasRole("admin") funciona com roles do Keycloak.

Testando

// Teste com JWT mockado
@Test
@WithMockUser(roles = "admin")
void adminEndpoint_withAdminRole_returns200() {
    mockMvc.perform(get("/admin/users"))
        .andExpect(status().isOk());
}

// Teste com JWT real (Spring Security Test)
@Test
void adminEndpoint_withJwt() {
    mockMvc.perform(get("/admin/users")
        .with(jwt().authorities(new SimpleGrantedAuthority("ROLE_admin")))
    ).andExpect(status().isOk());
}

🎯 Key Takeaway

Spring Boot + Keycloak = configuração OAuth2/OIDC padrão. Sem adapter proprietário. O único trabalho extra é o converter de roles — porque o Keycloak usa claims não-padrão para roles.

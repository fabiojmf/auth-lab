Backup e Recovery

O banco de dados do Keycloak contém tudo: usuários, realms, clients, roles, sessões. Perder esse banco é perder o acesso de todos os sistemas conectados. Backup não é opcional.

💾

Backup = banco de dados

O banco é a peça crítica. Todo o resto (binários, themes, config) é reproduzível. O banco não é.

O que fazer backup

✅ Fazer backup

Banco de dados — a peça crítica
Realm exports (JSON) — config-as-code
Custom themes/providers (JARs)
Certificados TLS
keycloak.conf / documentação de env vars

❌ NÃO precisa backup

Cache Infinispan (efêmero, reconstrói do DB)
Arquivos H2 (não devem existir em prod)
Binários do Keycloak (baixa de novo)
Logs antigos (já estão no log aggregator)

# Backup lógico (PostgreSQL) pg_dump -h db-host -U keycloak -d keycloak -F c -f backup_$(date +%Y%m%d).dump # Restore pg_restore -h db-host -U keycloak -d keycloak -c backup_20240115.dump # Backup automático diário (cron) 0 2 * * * pg_dump -h db -U keycloak -d keycloak -F c -f /backups/kc_$(date +\%Y\%m\%d).dump

Realm Export / Import

⚠️

Export NÃO é backup completo! O export de realm inclui hashes de senha, mas é lento e consome muita memória em realms grandes (10k+ usuários). Pode causar OOM ou timeout. Use para migrar configuração entre ambientes ou como backup complementar de realms pequenos — nunca como estratégia principal de backup. O backup do banco (pg_dump) é sempre a fonte primária.

# Export de realm (executar com KC parado ou em instância separada) kc.sh export --dir=/export --realm=meu-realm --users=realm_file # Import na inicialização (KC 20+) kc.sh start --import-realm --dir=/import # Import standalone (sem iniciar o servidor) kc.sh import --dir=/import

Disaster Recovery Plan

1
Defina RTO e RPO

RTO (Recovery Time Objective): quanto tempo pode ficar fora? RPO (Recovery Point Objective): quantos dados pode perder?

2
Automatize backups

Backup diário no mínimo. Para RPO baixo, use WAL archiving (point-in-time recovery).

3
Teste o restore regularmente

Um backup que nunca foi testado é o mesmo que não ter backup. Teste mensalmente.

Procedimento de Recovery

1
Restaurar banco de dados

pg_restore ou restore do snapshot cloud. Verificar integridade.

2
Iniciar Keycloak

O KC auto-migra o schema se necessário. Não precisa de ação manual.

3
Verificar

Health endpoint OK? Admin console acessível? Login de teste funciona? Então está recuperado.

🧪 Teste de Recovery — Checklist

• /health/ready retorna 200

• Admin console carrega e mostra realms

• Login com usuário de teste funciona

• Aplicações conectadas conseguem validar tokens

• Sessões ativas foram perdidas (esperado — estavam no cache)

🎯 Key Takeaway

Faça backup do BANCO DE DADOS. Todo o resto é reproduzível. Teste seu restore regularmente — backup não testado é backup inexistente. Defina RTO/RPO e automatize.

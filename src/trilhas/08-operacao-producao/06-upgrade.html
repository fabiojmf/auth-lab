Upgrade de Versão

Keycloak recebe releases trimestrais com patches de segurança, bug fixes e features novas. Ficar em versão antiga é acumular vulnerabilidades. Mas upgrade sem processo é pedir para quebrar produção.

⬆️

Regra de ouro

Backup → Teste em staging → Upgrade em produção. Nessa ordem. Sempre.

Regras de Upgrade

RegraDetalhe
Schema migra automaticamenteO Keycloak atualiza o banco no startup — sem scripts manuais
Upgrade sequencial recomendadoFaça N → N+1 → N+2. Pular versões não é oficialmente suportado e pode causar falhas na migração do schema
NÃO pode fazer downgradeSchema é forward-only. Rollback = restaurar backup do banco
Sempre leia o Upgrading GuideBreaking changes são documentadas por versão
⚠️

Downgrade impossível: uma vez que o Keycloak migra o schema do banco para a versão nova, não há como voltar. O único rollback é restaurar o backup do banco feito ANTES do upgrade.

Procedimento de Upgrade

1
Ler release notes + upgrading guide

Identifique breaking changes, features removidas, mudanças de comportamento.

2
Backup do banco de dados

ANTES de qualquer coisa. Este é seu rollback plan.

3
Testar em staging

Mesmo schema de banco, mesma config. Valide login flows, admin console, integrações.

4
Atualizar imagem / binário

Trocar a tag da imagem Docker ou substituir os binários.

5
Executar kc.sh build

Se houve mudança de configuração de build (providers, features, DB type).

6
Iniciar nova versão

O Keycloak detecta a versão do schema e migra automaticamente.

7
Verificar

Health endpoint, admin console, login de teste, logs sem erros.

8
Monitorar

Fique de olho nos logs e métricas por 24-48h após o upgrade.

Rolling Upgrade em Cluster

🔄 Kubernetes Rolling Update

• Rolling updates funcionam para patch/minor upgrades (ex: 24.0.1 → 24.0.5)

• Para major upgrades (ex: 23 → 24), prefira parar todas as instâncias antigas antes de subir as novas (recreate strategy)

• O primeiro pod novo que sobe executa a migração do schema

• Em major upgrades, pods antigos podem não funcionar com o schema novo — risco de erros

• Monitore durante o rollout — se houver erros, restaure o backup do banco e faça rollback da imagem

Problemas Comuns de Upgrade

ProblemaCausaSolução
Feature não funciona maisFeature deprecated foi removidaLer upgrading guide, usar alternativa
Custom SPI quebrouAPI interna mudouRecompilar contra a nova versão
Theme com erroTemplates FreeMarker mudaramAtualizar theme para nova estrutura
Client adapter não funcionaAdapters foram deprecatedMigrar para bibliotecas OIDC padrão

Migração WildFly → Quarkus

Se você ainda está no Keycloak legacy (WildFly), a migração para Quarkus é uma mudança significativa:

🦬 WildFly (Legacy)

Config: standalone.xml
CLI: jboss-cli.sh
Deploy: WAR em application server
Startup: 30-60 segundos
⚠️ End of life

🚀 Quarkus (Atual)

Config: keycloak.conf / env vars
CLI: kc.sh
Deploy: binário standalone / container
Startup: 5-15 segundos
✅ Ativamente mantido
# Guia oficial de migração: # https://www.keycloak.org/migration/migrating-to-quarkus # # Passos principais: # 1. Mapear standalone.xml → keycloak.conf (opções equivalentes) # 2. Converter jboss-cli scripts → kc.sh commands # 3. Atualizar Dockerfile (nova imagem base) # 4. Testar extensivamente em staging # 5. O banco de dados é compatível — não precisa migrar dados

🎯 Key Takeaway

Upgrades são seguros SE você fizer backup antes, testar em staging e ler o upgrading guide. Nunca faça upgrade direto em produção. O banco não faz downgrade — seu único rollback é o backup.

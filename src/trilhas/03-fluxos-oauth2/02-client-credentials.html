Client Credentials Flow

O fluxo mais simples. Não tem usuário, não tem browser, não tem redirect. É uma máquina se autenticando como ela mesma para acessar uma API.

Quando usar

  • Microserviço A chamando microserviço B
  • Job/cron que acessa uma API protegida
  • Integração entre sistemas (sem interação humana)
  • Backend que precisa de identidade própria para chamar APIs externas

Os participantes

💻 Client (Backend) Quer acessar uma API
🔐 Keycloak Valida e emite token
📦 API (Resource Server) Protege os dados

Note: não tem usuário. O token emitido representa a aplicação, não uma pessoa.

O token resultante

O token de um Client Credentials não tem um usuário associado. Olhe as diferenças:

Token de usuário (Auth Code)

  • sub: "user-123" (ID do usuário)
  • name: "Maria Silva"
  • email: "maria@empresa.com"
  • azp: "minha-app"

Token de service account

  • sub: "service-account-minha-app"
  • Sem name, sem email
  • azp: "minha-app"
  • Roles da service account

Testando com curl

Este é o fluxo mais fácil de testar. Um único request:

Nota: Client Credentials não retorna refresh_token. Não faz sentido — o client pode pedir um novo access_token a qualquer momento usando suas credenciais. Não precisa "renovar".

curl -X POST https://keycloak/realms/meu-realm/protocol/openid-connect/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=client_credentials" \
  -d "client_id=minha-app" \
  -d "client_secret=meu-secret"

💡 Resumo

Client Credentials = client_id + secret / token (sem usuário, sem browser)
Token representa = a aplicação, não uma pessoa
No Keycloak = "Contas de serviço habilitadas" = Sim
Quando usar = máquina-a-máquina, sem interação humana

🧪 Lab: Client Credentials na prática

Execute este fluxo no seu Keycloak local. Pré-requisito: ter rodado docker compose up -d na pasta docker/.

1
Obtenha um token com o client confidential: 
curl -X POST http://localhost:8080/realms/authlab/protocol/openid-connect/token \
  -d "grant_type=client_credentials" \
  -d "client_id=app-teste-confidential" \
  -d "client_secret=meu-secret-super-seguro"
2
Observe a resposta:

Você deve receber um JSON com access_token, expires_in e token_type. Note que NÃO há refresh_token (não faz sentido sem usuário).

3
Decodifique o token:

Copie o access_token e cole em jwt.io. Observe: o campo sub é o ID do service account (não de um humano), e azp é o client_id.

4
Teste com credenciais erradas: 
curl -X POST http://localhost:8080/realms/authlab/protocol/openid-connect/token \
  -d "grant_type=client_credentials" \
  -d "client_id=app-teste-confidential" \
  -d "client_secret=senha-errada"

Resultado esperado: {"error":"unauthorized_client"}

Authorization Code Flow

Este é o fluxo mais usado e mais seguro do OAuth2/OIDC. É o que acontece quando você clica "Entrar com Google" ou quando um sistema te redireciona para a tela de login do Keycloak.

Quando usar

  • Aplicações web com backend (Spring Boot, Node.js, Django)
  • SPAs com PKCE (React, Angular, Vue)
  • Qualquer cenário onde um humano precisa logar

Os participantes

👤 Usuário Quer acessar a aplicação
🌐 Browser Faz os redirects
💻 Aplicação (Client) Quer saber quem é o usuário
🔐 Keycloak (Auth Server) Autentica e emite tokens

O fluxo passo a passo

1 / 8

Por que o "code" intermediário?

Você pode se perguntar: por que não devolver o token direto no redirect? Por que esse passo extra do code?

❌ Token direto no redirect

  • Token aparece na URL do browser
  • Fica no histórico de navegação
  • Pode ser interceptado por extensões
  • Qualquer um que veja a URL tem o token

✅ Code + troca no backend

  • Code é de uso único e expira em segundos
  • Token é trocado no backend (server-to-server)
  • Token nunca aparece no browser
  • Troca exige client_secret (confidential)

E o PKCE?

Nota sobre o parâmetro state: nos passos 2 e 5 você viu um parâmetro state. Ele é uma proteção contra ataques CSRF — a aplicação gera um valor aleatório, envia ao Keycloak, e confere se volta igual. Se alguém tentar injetar um code falso, o state não vai bater.

Para clients public (SPAs, mobile) que não têm client_secret, existe o PKCE (Proof Key for Code Exchange). Ele adiciona uma proteção extra:

  1. O client gera um code_verifier aleatório
  2. Envia o hash dele (code_challenge) junto com o pedido de autorização
  3. Na hora de trocar o code por token, envia o code_verifier original
  4. O Keycloak confere: o hash bate? Se sim, emite o token

Isso garante que mesmo que alguém intercepte o code, não consegue trocá-lo por token sem o code_verifier original.

Erros comuns

Quando algo dá errado nesse fluxo, o Keycloak retorna um erro. Os mais frequentes:

  • invalid_grant — code expirado ou já usado
  • invalid_client — client_secret errado
  • redirect_uri mismatch — URI no request não bate com a configurada no Keycloak
  • unauthorized_client — fluxo não habilitado para esse client

💡 Resumo

Authorization Code Flow = redirect / login no Keycloak / code / troca por token no backend
Com PKCE = mesma coisa, mas o code_verifier protege a troca (para clients sem secret)
Quando usar = sempre que um humano precisa logar. É o padrão.

🧪 Lab: Authorization Code Flow na prática

Simule o fluxo completo usando seu browser e curl.

1
Abra esta URL no browser: 
http://localhost:8080/realms/authlab/protocol/openid-connect/auth
  ?response_type=code
  &client_id=app-teste-confidential
  &redirect_uri=http://localhost:3000/callback
  &scope=openid profile email
  &state=teste123

Você será redirecionado para a tela de login do Keycloak.

2
Faça login com maria/senha123

Após o login, o browser tentará redirecionar para http://localhost:3000/callback?code=XXXXX&state=teste123. Como não temos servidor na 3000, vai dar erro — mas copie o code da URL.

3
Troque o code por tokens: 
curl -X POST http://localhost:8080/realms/authlab/protocol/openid-connect/token \
  -d "grant_type=authorization_code" \
  -d "client_id=app-teste-confidential" \
  -d "client_secret=meu-secret-super-seguro" \
  -d "redirect_uri=http://localhost:3000/callback" \
  -d "code=COLE_O_CODE_AQUI"
4
Tente reusar o code:

Execute o mesmo curl novamente. Resultado: {"error":"invalid_grant"} — o code é de uso único!

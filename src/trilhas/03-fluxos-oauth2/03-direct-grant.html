Direct Grant (Resource Owner Password)

Neste fluxo, a aplicação recebe o usuário e senha diretamente e manda para o Keycloak trocar por token. Sem redirect, sem tela do Keycloak.

⚠️ Atenção

Este fluxo foi removido do OAuth 2.1 (a versão mais recente do protocolo). A aplicação vê a senha do usuário, o que é um risco. Use apenas quando não há alternativa (CLIs, sistemas legados, testes). Para novos projetos, prefira sempre o Authorization Code Flow.

Quando usar

  • CLIs e ferramentas de linha de comando
  • Sistemas legados que não suportam redirect
  • Testes rápidos (curl)
  • Sistemas que têm sua própria tela de login e não podem redirecionar

Os participantes

👤 Usuário Fornece suas credenciais
💻 Aplicação (Client) Recebe a senha e repassa
🔐 Keycloak Valida credenciais e emite token

O fluxo passo a passo

1 / 3

Por que é menos seguro?

✅ Authorization Code

  • Aplicação nunca vê a senha
  • Suporta MFA naturalmente
  • Suporta login social
  • Usuário loga no Keycloak (confiável)

⚠️ Direct Grant

  • Aplicação recebe a senha
  • MFA é difícil de implementar
  • Não suporta login social
  • Usuário confia a senha à aplicação

Testando com curl

curl -X POST https://keycloak/realms/meu-realm/protocol/openid-connect/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=password" \
  -d "client_id=minha-app" \
  -d "client_secret=meu-secret" \
  -d "username=maria" \
  -d "password=senha123"

Alternativa moderna para CLIs: se a CLI pode abrir um browser no computador do usuário, considere o Device Authorization Grant (RFC 8628). É o que ferramentas como az login e aws sso login usam — mais seguro que Direct Grant porque a senha nunca passa pela aplicação.

💡 Resumo

Direct Grant = username + password + client_id → token (sem redirect)
A aplicação vê a senha — por isso é menos seguro
No Keycloak = "Grants de Acesso direto habilitado" = Sim
Quando usar = CLIs, legado, testes. Evite em produção se possível.

🧪 Lab: Direct Grant na prática

Teste o fluxo Resource Owner Password com o usuário pré-configurado.

1
Obtenha um token com usuário/senha: 
curl -X POST http://localhost:8080/realms/authlab/protocol/openid-connect/token \
  -d "grant_type=password" \
  -d "client_id=app-teste-confidential" \
  -d "client_secret=meu-secret-super-seguro" \
  -d "username=maria" \
  -d "password=senha123"
2
Compare com o Client Credentials:

Agora você recebe access_token + refresh_token + id_token. Decodifique o access_token — o campo sub agora é o ID da Maria, e preferred_username mostra "maria".

3
Teste com senha errada: 
curl -X POST http://localhost:8080/realms/authlab/protocol/openid-connect/token \
  -d "grant_type=password" \
  -d "client_id=app-teste-confidential" \
  -d "client_secret=meu-secret-super-seguro" \
  -d "username=maria" \
  -d "password=errada"

Resultado: {"error":"invalid_grant","error_description":"Invalid user credentials"}

4
Teste com o usuário João (menos permissões):

Repita o passo 1 com username=joao e password=senha123. Compare as roles no token — João tem apenas "user" e "viewer", enquanto Maria tem "admin".

Refresh Token Flow

O access_token tem vida curta (minutos). Quando expira, o usuário precisaria logar de novo. O refresh_token resolve isso — permite renovar o acesso sem interação do usuário.

O ciclo de vida dos tokens

access_token 5–30 min
refresh_token horas / dias
Login
Access expira
Refresh expira
→ login de novo

O fluxo passo a passo

1 / 4

Por que não usar um access_token de longa duração?

❌ Access token longo (ex: 24h)

  • Se roubado, atacante tem acesso por 24h
  • Não pode ser revogado facilmente (é stateless)
  • Mudanças de permissão demoram a refletir

✅ Access curto + Refresh longo

  • Se access roubado, expira em minutos
  • Refresh pode ser revogado no servidor
  • A cada renovação, permissões são atualizadas

Quando o refresh_token também expira

Se o refresh_token expirar (ou for revogado), o usuário precisa fazer login novamente. Isso acontece quando:

  • O tempo de vida do refresh_token acabou
  • O administrador revogou a sessão do usuário
  • O usuário fez logout
  • As chaves do realm foram rotacionadas (isso invalida todos os tokens existentes)

Refresh Token Rotation (segurança moderna)

Uma prática recomendada: a cada vez que o refresh_token é usado, o Keycloak emite um novo refresh_token e invalida o anterior. Isso significa que:

  • Cada refresh_token só pode ser usado uma vez
  • Se um atacante roubar um refresh_token antigo, ele já foi invalidado
  • Se o Keycloak detectar reuso de um refresh_token já usado, revoga toda a sessão (possível roubo)

O Keycloak suporta isso nativamente. É configurável por realm em "Realm Settings" / "Tokens" / "Revoke Refresh Token".

Onde guardar o refresh_token?

O refresh_token é sensível — se alguém roubar, pode gerar novos access_tokens. Guarde com cuidado:

  • Backend (melhor): em memória ou banco de dados da sessão do servidor
  • SPA: idealmente no backend (padrão BFF). Se no browser, apenas em memória — nunca em localStorage
  • Mobile: secure storage do sistema (iOS Keychain / Android Keystore)

Testando com curl

curl -X POST https://keycloak/realms/meu-realm/protocol/openid-connect/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=refresh_token" \
  -d "client_id=minha-app" \
  -d "client_secret=meu-secret" \
  -d "refresh_token=eyJhbG..."

💡 Resumo

Refresh Token = troca um refresh_token por um novo access_token
Por que existe = access_token curto (segurança) + refresh_token longo (conveniência)
Quando falha = refresh expirado, sessão revogada, chaves rotacionadas / login de novo

🧪 Lab: Refresh Token na prática

Use o refresh_token obtido no lab anterior para renovar o access_token.

1
Primeiro, obtenha tokens via Direct Grant: 
curl -X POST http://localhost:8080/realms/authlab/protocol/openid-connect/token \
  -d "grant_type=password" \
  -d "client_id=app-teste-confidential" \
  -d "client_secret=meu-secret-super-seguro" \
  -d "username=maria" \
  -d "password=senha123"

Copie o refresh_token da resposta.

2
Use o refresh_token para obter um novo access_token: 
curl -X POST http://localhost:8080/realms/authlab/protocol/openid-connect/token \
  -d "grant_type=refresh_token" \
  -d "client_id=app-teste-confidential" \
  -d "client_secret=meu-secret-super-seguro" \
  -d "refresh_token=COLE_O_REFRESH_TOKEN_AQUI"

Você recebe um NOVO access_token (e possivelmente um novo refresh_token via rotation).

3
Compare os tokens:

Decodifique ambos os access_tokens em jwt.io. Note que o iat (issued at) e exp são diferentes — é um token novo com nova validade.

Infrastructure as Code (Terraform)

Gerenciar Keycloak manualmente pelo Admin Console funciona para um ambiente. Para múltiplos ambientes com consistência, rastreabilidade e reprodutibilidade, você precisa de Infrastructure as Code.

📋

Por que IaC para Keycloak?

Reprodutibilidade, versionamento no Git, audit trail, consistência entre dev/staging/prod e rollback fácil.

Terraform Keycloak Provider

O provider oficial keycloak/keycloak (mantido pelo projeto Keycloak desde 2025, anteriormente mrparkers/keycloak) permite gerenciar via Terraform:

RecursoTerraform Resource
Realmskeycloak_realm
Clients (OIDC/SAML)keycloak_openid_client
Client Scopeskeycloak_openid_client_scope
Roleskeycloak_role
Groupskeycloak_group
Identity Providerskeycloak_oidc_identity_provider
User Federationkeycloak_ldap_user_federation
Auth Flowskeycloak_authentication_flow

Exemplo Básico

terraform { required_providers { keycloak = { source = "keycloak/keycloak" version = "~> 5.0" } } } provider "keycloak" { client_id = "admin-cli" username = var.kc_admin_user password = var.kc_admin_password url = var.kc_url } resource "keycloak_realm" "my_realm" { realm = "my-app" enabled = true login_theme = "my-custom-theme" default_signature_algorithm = "RS256" } resource "keycloak_openid_client" "frontend" { realm_id = keycloak_realm.my_realm.id client_id = "my-frontend" access_type = "PUBLIC" standard_flow_enabled = true valid_redirect_uris = ["https://app.example.com/*"] web_origins = ["https://app.example.com"] }

Workflow

1

terraform init — baixa o provider

2

terraform plan — mostra o que será criado/alterado/destruído

3

terraform apply — aplica as mudanças no Keycloak

4

terraform import — importa recursos existentes para o state

Boas Práticas

✅ Faça

State remoto (S3, GCS, Terraform Cloud)
Variáveis para secrets — nunca hardcode
Ambientes separados (workspaces ou dirs)
Import antes de gerenciar KC existente
Pin de versão do provider

❌ Evite

State local em produção
Senhas no código .tf
Um único state para tudo
Aplicar sem plan review
Gerenciar usuários individuais via TF

Limitações

⚠️

Nem todos os recursos do Keycloak são suportados pelo provider. Custom SPIs, themes e algumas configurações avançadas precisam ser gerenciados por outros meios (scripts, Dockerfile).

  • Alguns recursos exigem ordenação específica (realm antes de client)
  • Mudanças manuais no Admin Console causam drift no state
  • Provider pode demorar para suportar features novas do KC

Alternativa: Keycloak Config CLI

O adorsys/keycloak-config-cli usa YAML/JSON para configurar o Keycloak. Mais simples que Terraform para quem não precisa de state management:

realm: my-app enabled: true clients: - clientId: my-frontend publicClient: true redirectUris: - "https://app.example.com/*"

🎯 Key Takeaway

Terraform dá reprodutibilidade e versionamento à configuração do Keycloak. Essencial para multi-ambiente. Use variáveis para secrets, state remoto, e sempre faça terraform plan antes de aplicar. Para casos mais simples, considere o Keycloak Config CLI.

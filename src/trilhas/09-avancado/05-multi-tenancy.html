Multi-Tenancy

Multi-tenancy é servir múltiplas organizações (tenants) a partir de uma única instância Keycloak. Não existe solução perfeita — cada abordagem tem trade-offs.

🏢

O Dilema Central

Isolamento completo (seguro, mas caro) vs recurso compartilhado (eficiente, mas complexo)?

Abordagem 1: Um Realm por Tenant

✅ Vantagens

Isolamento completo (users, clients, config)
Cada tenant tem login page própria
Branding independente por tenant
IDPs diferentes por tenant
Configuração simples e intuitiva

❌ Desvantagens

Escala limitada (~100-500 realms)
Cada realm consome memória/CPU
Overhead de gerenciamento
Automação obrigatória (Admin API/Terraform)
Monitoramento mais complexo

Abordagem 2: Realm Único com Groups/Attributes

✅ Vantagens

Escala para milhares de tenants
Menor consumo de recursos
Gerenciamento centralizado
Um único ponto de configuração

❌ Desvantagens

Sem isolamento real entre tenants
Autorização complexa na aplicação
Login page compartilhada
Risco de vazamento entre tenants

Nesta abordagem, o tenant é identificado por um user attribute ou group, e incluído no token via protocol mapper:

// Token com claim de tenant (via mapper) { "sub": "user-123", "tenant_id": "acme-corp", "realm_access": { "roles": ["user"] } } // A aplicação usa tenant_id para filtrar dados

Abordagem 3: Organizations (Keycloak 25+)

🧪

Introduzido no Keycloak 25 (preview), GA no Keycloak 26. Organizations oferece suporte nativo a multi-tenancy dentro de um realm — membros, IDPs por organização e domínios vinculados. Desde KC 26 é uma feature totalmente suportada.

Como Escolher?

NecessidadeAbordagem Recomendada
Isolamento completo obrigatórioRealm por tenant
1000+ tenantsRealm único (realms não escalam tanto)
IDPs diferentes por tenantRealm por tenant ou Organizations
SaaS simples, tenant = atributoRealm único com groups
Branding diferente por tenantRealm por tenant (theme por realm)
Compliance/regulatório rígidoRealm por tenant (ou instâncias separadas)

Patterns: Realm por Tenant

  • Criação dinâmica: Admin API cria realm no onboarding do tenant
  • Theme compartilhado: um theme com branding dinâmico via realm attributes
  • Roteamento: subdomínio / realm no reverse proxy (acme.auth.example.com / realm acme)
  • Terraform: módulo reutilizável que cria realm + clients + roles padrão

Patterns: Realm Único

  • Authenticator customizado: SPI que seleciona tenant no login (por domínio do email, por exemplo)
  • Protocol Mapper: inclui tenant_id no token baseado em group ou attribute
  • Autorização na app: middleware que filtra dados pelo tenant_id do token
  • Groups hierárquicos: /tenants/acme/admins, /tenants/acme/users

🎯 Key Takeaway

Não existe solução perfeita de multi-tenancy. Realm por tenant dá isolamento mas não escala infinitamente. Realm único escala mas exige autorização complexa na aplicação. Escolha baseado em: necessidade de isolamento, número de tenants e budget de complexidade.

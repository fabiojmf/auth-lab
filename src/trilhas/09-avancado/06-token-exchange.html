Token Exchange

Token Exchange permite trocar um token por outro com características diferentes — audience, escopo ou identidade. É definido pela RFC 8693 e resolve problemas reais em arquiteturas de microserviços.

🔄

O Problema

Serviço A tem um token do usuário, mas precisa chamar Serviço B que aceita apenas tokens com audience diferente. Como obter o token certo sem pedir login novamente?

ℹ️

Duas versões coexistem no Keycloak. O Standard Token Exchange V2 é habilitado por padrão e totalmente suportado (apenas internal-to-internal). O Legacy Token Exchange V1 (impersonation, external tokens) é preview/deprecated e requer --features=token-exchange + --features=admin-fine-grained-authz.

Casos de Uso

CasoDescriçãoVersão
Audience swapToken para serviço diferente no mesmo realmV2 (suportado)
DownscopingToken com menos permissões/audiencesV2 (suportado)
DelegaçãoAgir em nome do usuário em outro serviçoV1 (legacy)
ImpersonationAdmin obtém token como outro usuárioV1 (legacy)
External tokenTrocar token de IDP externo por token KCV1 (legacy) / JWT Auth Grant

Grant Type

grant_type=urn:ietf:params:oauth:grant-type:token-exchange

Parâmetros da Request

ParâmetroDescriçãoObrigatório
subject_tokenO token que você temSim
subject_token_typeurn:ietf:params:oauth:token-type:access_tokenSim
requested_token_typeTipo do token desejadoNão
audienceClient/serviço de destinoNão*
scopeEscopos desejados no novo tokenNão

Exemplo Completo

# Serviço A troca seu token por um token válido para Serviço B curl -X POST \ https://keycloak.example.com/realms/my-app/protocol/openid-connect/token \ -d "client_id=service-a" \ -d "client_secret=secret-a" \ -d "grant_type=urn:ietf:params:oauth:grant-type:token-exchange" \ -d "subject_token=eyJhbGciOiJSUzI1NiIs..." \ -d "subject_token_type=urn:ietf:params:oauth:token-type:access_token" \ -d "audience=service-b" # Resposta: novo access_token com audience=service-b { "access_token": "eyJhbG...(novo token)...", "token_type": "Bearer", "expires_in": 300 }

Configuração: Standard Token Exchange (V2)

O V2 é habilitado por padrão no KC 26+. Apenas troca internal-to-internal (audience swap entre clients do mesmo realm).

1

No client que fará o exchange (ex: service-a): ativar o switch Standard token exchange na aba Settings

2

O client deve ser confidential (não público)

3

O subject_token enviado deve ter o client solicitante como audience (aud)

Não requer fine-grained admin permissions nem feature flags adicionais.

Configuração: Legacy Token Exchange (V1)

Para impersonation, external-to-internal ou internal-to-external, use o V1 (preview/deprecated):

1

Habilitar features: kc.sh start --features=token-exchange,admin-fine-grained-authz

2

No client de destino: Permissions → habilitar → configurar policy token-exchange

3

Criar Client Policy que permite o client de origem fazer exchange

4

Associar policy à permission "token-exchange" do client de destino

⚠️

V1 é deprecated. Será removido em versões futuras. Para internal-to-internal, use V2. Para external token exchange, considere JWT Authorization Grant (RFC 7523) + V2 como alternativa suportada.

⚠️ Segurança

🔒

Token exchange mal configurado = escalação de privilégios. Se qualquer client puder trocar tokens para qualquer outro, um atacante com acesso a um serviço compromete todos os outros.

  • Restrinja quais clients podem fazer exchange para quais targets
  • Audite eventos de token exchange (Event Listener)
  • Use escopos mínimos no token resultante
  • Monitore volume anormal de exchanges

Quando NÃO usar Token Exchange

SituaçãoAlternativa mais simples
Precisa apenas de audience diferenteConfigure audience mapper no client
Comunicação serviço-a-serviço sem usuárioUse client_credentials grant
Contexto do usuário no downstreamPasse o token original (se audience permite)
Apenas precisa de menos scopesSolicite menos scopes no login inicial

🎯 Key Takeaway

Token exchange resolve problemas reais em microserviços — trocar tokens entre serviços com audiences diferentes, delegação e impersonation. Mas adiciona complexidade e risco de segurança. Use apenas quando alternativas mais simples (audience mapper, client_credentials) não resolvem.

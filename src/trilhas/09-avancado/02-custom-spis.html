Custom SPIs (Service Provider Interfaces)

SPIs são os pontos de extensão do Keycloak. Quando as funcionalidades built-in não resolvem seu problema, você pode escrever código Java que roda dentro do Keycloak.

🔌

Quando usar SPIs?

Quando o Keycloak não oferece nativamente o que você precisa — e nenhuma configuração ou workaround resolve.

Casos de Uso Comuns

SPIProblema que resolveExemplo
User StorageUsuários em banco/API legadoValidar login contra API REST externa
AuthenticatorPasso customizado no loginVerificar IP, validar CPF, 2FA custom
Event ListenerReagir a eventos do KCEnviar login para SIEM/Kafka
Protocol MapperClaims customizados no tokenBuscar dados de API e incluir no JWT
REST ResourceEndpoint customizado na APIExpor dados específicos via REST

Arquitetura de um SPI

┌─────────────────────────────────────────┐ │ Keycloak Runtime │ │ │ │ ┌─────────────┐ ┌────────────────┐ │ │ │ SPI │ │ Seu Provider │ │ │ │ Interface │◄──│ (implements) │ │ │ └─────────────┘ └────────────────┘ │ │ ▲ │ │ ┌─────────────────────────┐ │ │ │ ProviderFactory │ │ │ │ (cria instâncias) │ │ │ └─────────────────────────┘ │ │ │ │ META-INF/services/ │ │ └── org.keycloak.storage. │ │ UserStorageProviderFactory │ └─────────────────────────────────────────┘

Cada SPI tem 3 componentes:

  • Provider Interface — o contrato que você implementa
  • ProviderFactory — cria instâncias do seu provider
  • META-INF/services/ — arquivo que registra sua factory

User Storage SPI (mais comum)

Permite que o Keycloak autentique usuários contra uma fonte externa:

public class MeuStorageProvider implements UserStorageProvider, UserLookupProvider, CredentialInputValidator { @Override public UserModel getUserByUsername(RealmModel realm, String username) { // Buscar usuário na sua API/banco ExternalUser ext = minhaApi.findByUsername(username); if (ext == null) return null; return new UserAdapter(session, realm, model, ext); } @Override public boolean isValid(RealmModel realm, UserModel user, CredentialInput input) { // Validar senha contra fonte externa return minhaApi.validatePassword( user.getUsername(), input.getChallengeResponse() ); } }

📖 READ_ONLY

Apenas valida credenciais
Usuários não podem trocar senha via KC
Fonte externa é a verdade
Mais simples de implementar

✏️ WRITABLE

Valida e também cria/atualiza
Sincroniza mudanças de volta
Implementa UserRegistrationProvider
Mais complexo e arriscado

Deploy de SPIs

1

Build do JAR: mvn package (ou Gradle)

2

Copiar para /opt/keycloak/providers/meu-spi.jar

3

Executar kc.sh build (registra o provider)

4

Restart do Keycloak — provider disponível no Admin Console

Workflow de Desenvolvimento

Use start-dev para hot-reload de providers durante desenvolvimento. Para testes automatizados, use Keycloak Testcontainers:

@Testcontainers class MeuProviderTest { @Container static KeycloakContainer kc = new KeycloakContainer("quay.io/keycloak/keycloak:latest") .withProviderClassesFrom("target/classes"); }

⚠️ Riscos

⚠️

SPIs rodam DENTRO da JVM do Keycloak. Um bug no seu código pode derrubar o servidor inteiro. Um deadlock pode travar todos os logins. Uma exception não tratada pode bloquear autenticação para todos os usuários.

RiscoMitigação
Crash do servidorTry-catch em todo ponto de entrada, fallback graceful
Thread safetyProviders são criados por request — não compartilhe estado mutável
PerformanceTimeouts em chamadas externas, circuit breaker
Acoplamento de versãoSPI API muda entre versões — teste a cada upgrade

🎯 Key Takeaway

SPIs são poderosos mas arriscados. Rodam dentro do Keycloak — bugs podem derrubar o servidor. Use apenas quando funcionalidades built-in realmente não resolvem o problema. Sempre implemente com tratamento de erros robusto e testes automatizados.

Social Login

Social Login é um caso específico de Identity Brokering: o Keycloak vem com provedores sociais pré-configurados, simplificando a integração com Google, GitHub, Facebook e outros.

Provedores built-in no Keycloak

🔵 Google
⚫ GitHub
🔵 Facebook
🟦 Microsoft
⚪ Apple
🔵 Twitter/X
🔵 LinkedIn
🟣 GitLab
🟠 Stack Overflow
🔵 Bitbucket
🟡 PayPal
⚙️ OpenID Connect (genérico)

Cada um já tem os endpoints pré-configurados. Você só precisa fornecer Client ID + Secret. No Admin Console do KC 20+, todos ficam em Identity Providers / Add provider (não existe seção separada "Social").

Configuração passo a passo — Google como exemplo

☁️

1. No Google Cloud Console: APIs & Services / Credentials / Create OAuth Client ID

Tipo: Web Application. Anote o Client ID e Client Secret gerados.

🔗

2. No Google, configure o Authorized Redirect URI:

https://seu-keycloak.com/realms/{realm}/broker/google/endpoint
⚙️

3. No Keycloak: Identity Providers / Add provider / Google

Cole o Client ID e Client Secret do passo 1.

📋

4. Configure os scopes: email e profile (padrão)

Scopes adicionais: openid, phone, address (se necessário).

✅

5. Salve. O botão "Login with Google" aparece automaticamente na tela de login.

Teste: acesse a tela de login do realm e clique no botão.

Account Linking

Um usuário pode ter múltiplas formas de login vinculadas à mesma conta:

Cenário Como funciona
Usuário com senha quer adicionar Google Acessa Account Console / Linked Accounts / vincula a conta Google
Usuário logou via Google, quer definir senha Acessa Account Console / Password / define uma senha local
Primeiro login via Google, email já existe First Login Flow decide: vincular automaticamente ou pedir confirmação

Mappers de Social Login

Cada provedor social retorna claims diferentes. Mappers transformam esses claims em atributos do Keycloak:

Google: picture → KC: avatar_url (custom attr)
GitHub: login → KC: github_username (custom attr)
Microsoft: preferred_username → KC: email (standard attr)

Configure em: Identity Providers / [provider] / Mappers / Add mapper

Segurança — riscos do Social Login

⚠️ Considerações de segurança

Email verification trust

O Google diz que maria@gmail.com é verificado. Mas e se alguém criou uma conta Google com o email corporativo de outra pessoa?

Configuração: "Trust Email" no IDP — habilite apenas para provedores confiáveis.

Account takeover via email

Atacante cria conta social com email da vítima / First Login Flow vincula à conta existente / atacante tem acesso.

Mitigação: exigir verificação de email antes de vincular (Verify Existing Account no First Login Flow).

Scope creep

Não peça mais scopes do que precisa. email + profile é suficiente para login. Evite pedir acesso a contatos, calendário, etc.

Quando usar — e quando NÃO usar

Contexto Social Login? Por quê
App B2C (e-commerce, SaaS) ✓ Sim Reduz fricção, aumenta conversão, UX familiar
App interna corporativa ✗ Não Use o IDP corporativo (Azure AD, LDAP). Controle centralizado.
Sistema judicial / governo ✗ Não Requisitos de segurança, auditoria, certificação digital
Plataforma de desenvolvedores ✓ Sim GitHub login é esperado e natural para devs
App com dados sensíveis (saúde, financeiro) ⚠ Depende Pode usar para onboarding, mas exija MFA adicional

💡 Ponto-chave

Social Login é apenas Identity Brokering com provedores pré-configurados. Tecnicamente, "Login com Google" funciona exatamente como "Login com outro Keycloak" — a diferença é que o Google já vem como template pronto. A decisão de usar ou não é de negócio e segurança, não técnica.

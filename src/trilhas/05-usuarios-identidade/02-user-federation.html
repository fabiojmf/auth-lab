User Federation

User Federation é o mecanismo que permite ao Keycloak consultar uma base de usuários externa — sem duplicar dados. A empresa já tem 10.000 usuários no Active Directory? O Keycloak se conecta lá e valida credenciais diretamente.

Por que federar?

❌ Sem Federation

  • Duplicar milhares de usuários no Keycloak
  • Manter senhas sincronizadas manualmente
  • Usuário troca senha no AD / Keycloak não sabe
  • Dois pontos de gestão de identidade

✅ Com Federation

  • Keycloak consulta o AD/LDAP em tempo de login
  • Senha validada na fonte original
  • Usuário troca senha no AD / funciona imediatamente
  • Um ponto de gestão, Keycloak só autentica

Como funciona

👤

1. Usuário digita login e senha na tela do Keycloak

🔍

2. Keycloak não encontra o usuário localmente

📡

3. Keycloak consulta o provider de federation (LDAP/AD) por ordem de prioridade

🔐

4. LDAP valida: "Sim, esse usuário existe e a senha está correta"

✅

5. Keycloak cria/atualiza o usuário local (cache) e emite o token

O usuário aparece no Keycloak, mas a fonte de verdade continua sendo o LDAP.

Tipos de Federation Providers

📂 LDAP / Active Directory

Built-in no Keycloak

  • Configuração via Admin Console
  • Suporta LDAP v3 e AD
  • Mappers para atributos, roles, grupos
  • Sync periódico configurável

⚙️ Custom User Storage SPI

Código Java customizado

  • Conecta a qualquer fonte (DB, REST API, legado)
  • Implementa interfaces do Keycloak
  • Deploy como JAR no servidor
  • Ex: PJEStorageProvider para bases judiciais

Configuração LDAP — parâmetros essenciais

Parâmetro Exemplo Descrição
Vendor Active Directory ou Other Tipo de servidor LDAP — auto-configura defaults (UUID attr, RDN, User Object Classes)
Connection URL ldaps://ad.empresa.com:636 Endereço do servidor LDAP (389 = plain, 636 = TLS)
Bind DN cn=keycloak,ou=services,dc=empresa,dc=com Conta que o Keycloak usa para consultar o LDAP
Bind Credential ******** Senha da conta de bind
Users DN ou=users,dc=empresa,dc=com Base de busca — onde procurar usuários
User Object Classes inetOrgPerson, organizationalPerson Filtro de classes LDAP para identificar usuários
Username LDAP Attribute uid ou sAMAccountName Qual atributo LDAP mapeia para o username
UUID LDAP Attribute entryUUID ou objectGUID Identificador único imutável do registro LDAP (entryUUID para LDAP, objectGUID para AD)
RDN LDAP Attribute uid ou cn Atributo usado como RDN (Relative Distinguished Name) nas entradas LDAP

Edit Mode — como o Keycloak interage com o LDAP

READ_ONLY

Keycloak só lê do LDAP. Usuário não pode trocar senha pelo Keycloak. Alterações só no LDAP diretamente.

Mais seguro

WRITABLE

Keycloak pode escrever no LDAP. Troca de senha, atualização de atributos — tudo reflete no LDAP.

Bidirecional

UNSYNCED

Alterações ficam só no Keycloak local. LDAP não é atualizado. Útil para testes ou atributos extras.

Independente

Import Users e Sync

Configuração Sim Não
Import Users Cria cópia local no KC (cache). Login mais rápido. Consulta LDAP a cada login. Sempre atualizado.
Periodic Full Sync Sincroniza todos os usuários periodicamente Só importa no primeiro login de cada usuário
Periodic Changed Users Sync Sincroniza apenas alterações recentes Alterações só aparecem no próximo login

Mappers LDAP

Mappers definem como atributos do LDAP são traduzidos para o Keycloak:

user-attribute-ldap-mapper

Mapeia um atributo LDAP / atributo do usuário no KC.
Ex: mail / email, cn / firstName

role-ldap-mapper

Mapeia roles do LDAP / roles no Keycloak.
Ex: role LDAP cn=admins,ou=roles / role KC admin

group-ldap-mapper

Mapeia OUs ou grupos LDAP / groups no Keycloak.
Ex: ou=TI / group /TI

msad-user-account-control-mapper

Específico para Active Directory: mapeia flags de conta (enabled/disabled, password expired) do atributo userAccountControl.

Prioridade entre providers

Quando existem múltiplos federation providers configurados, o campo Priority determina a ordem de busca. O Keycloak consulta do menor número (maior prioridade) para o maior. Se encontrar o usuário no primeiro provider, para ali.

Custom User Storage SPI

Para fontes que não são LDAP (banco de dados legado, API REST, sistema proprietário), organizações podem implementar a interface UserStorageProvider em Java. O provider é empacotado como JAR e deployado no Keycloak.

public class MeuStorageProvider implements
    UserStorageProvider,
    UserLookupProvider,
    CredentialInputValidator {

    // Buscar usuário por username
    public UserModel getUserByUsername(RealmModel realm, String username) { ... }

    // Validar credenciais contra a fonte externa
    public boolean isValid(RealmModel realm, UserModel user,
                           CredentialInput input) { ... }
}

Exemplo real: PJEStorageProvider conecta o Keycloak à base de usuários do sistema judicial.

Cenário real: troubleshooting

🐛 "Usuário existe no LDAP mas não consegue logar no Keycloak"

  1. Verifique se o federation provider está ativo — pode estar desabilitado
  2. Confira o Users DN — o usuário pode estar em outra OU não coberta pela busca
  3. Teste a conexão — botão "Test Connection" e "Test Authentication" no Admin Console
  4. Verifique o Edit Mode — se READ_ONLY e a senha expirou no LDAP, o usuário não consegue trocar via KC
  5. Cheque o User Object Classes — se o filtro não inclui a classe do usuário, ele é invisível
  6. Olhe os logs — org.keycloak.storage em DEBUG mostra as queries LDAP

💡 Ponto-chave

User Federation = o Keycloak pergunta ao sistema externo "esse usuário existe e a senha está correta?" no momento do login. A fonte de verdade dos usuários continua sendo o LDAP/AD — o Keycloak é apenas o intermediário que centraliza a autenticação.

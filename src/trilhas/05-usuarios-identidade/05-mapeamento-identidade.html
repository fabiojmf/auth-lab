Mapeamento e Fluxo de Identidade

A identidade no Keycloak é construída em camadas. Independente de onde o usuário vem (local, LDAP ou IDP externo), os dados passam por transformações até se tornarem claims no token final. Entender esse fluxo é essencial para configurar corretamente.

O ciclo de vida da identidade

🚪 Origem

Usuário chega (local, federado ou via IDP)

→
🔄 Mapeamento

Atributos são mapeados e transformados

→
🛡️ Roles

Roles atribuídas (direta, grupo ou mapper)

→
🎫 Token

Token construído com a identidade final

Fluxo de atributos por origem

Cada tipo de origem tem um caminho diferente até o token:

mail Atributo LDAP
LDAP Mapper →
email User Attribute (KC)
Protocol Mapper →
email Claim no Token

LDAP Mapper (user-attribute-ldap-mapper) roda na sincronização/login. Protocol Mapper roda na geração do token.

preferred_username Claim do IDP
IDP Mapper →
username User Attribute (KC)
Protocol Mapper →
preferred_username Claim no Token

IDP Mapper roda no login via IDP (First Login Flow ou logins subsequentes). Protocol Mapper roda na geração do token.

cpf = 123.456.789-00 User Attribute (KC)
Protocol Mapper →
"cpf": "123.456.789-00" Claim no Token

Usuários locais não precisam de mapper de origem — o atributo já está no KC. Só precisa do Protocol Mapper para incluir no token.

IDP Mappers vs Protocol Mappers — são coisas diferentes!

🔀 IDP Mapper

Dados ENTRAM no Keycloak

  • Roda quando usuário loga via IDP externo
  • Transforma claims do IDP / atributos do KC
  • Configurado em: Identity Providers / Mappers
  • Ex: claim "department" do Google / attr "departamento" no KC
  • Executa: no login (First Login + logins subsequentes)
  • Sync Mode: INHERIT, IMPORT (só no primeiro login), LEGACY ou FORCE (sempre sobrescreve)

🎫 Protocol Mapper

Dados SAEM do Keycloak

  • Roda quando o token é gerado
  • Transforma dados do KC / claims no token
  • Configurado em: Client Scopes / Mappers
  • Ex: attr "cpf" do KC / claim "cpf" no access_token
  • Executa: a cada emissão/refresh de token

First Login Flow — em profundidade

O First Login Flow é uma sequência de passos executados quando um usuário de IDP externo faz login pela primeira vez. Cada passo é um authenticator configurável:

📝

Review Profile

Mostra formulário para o usuário revisar/completar dados (nome, email). Útil quando o IDP não fornece todas as informações necessárias.

Configuração: REQUIRED (sempre mostra), ALTERNATIVE (só se faltar dados), DISABLED

➕

Create User If Unique

Cria automaticamente um usuário local no Keycloak com os dados recebidos do IDP — mas apenas se não existir outro com o mesmo email/username. Sem isso, o login falha.

Quase sempre REQUIRED — sem usuário local, não há como emitir token.

🔗

Link Existing Account

Se já existe um usuário com o mesmo email, tenta vincular em vez de criar duplicata. Evita o erro "User already exists".

Configuração: ALTERNATIVE — só executa se encontrar conta existente.

📧

Verify Existing Account

Antes de vincular, envia email de confirmação ao dono da conta existente. Previne account takeover.

Segurança: REQUIRED quando "Trust Email" está desabilitado no IDP.

Account Linking

Account Linking conecta múltiplas identidades a um único usuário no Keycloak:

Cenário Fluxo Resultado
Usuário local quer vincular Google Account Console / Linked Accounts / Google / Link Pode logar com senha OU Google
Usuário de IDP quer vincular conta local First Login Flow detecta email existente / vincula Mesma conta, duas formas de login
Conflito: email já existe Depende do First Login Flow configurado Vincula, pede verificação, ou falha

Resolução de conflitos

⚠️ O que acontece quando o email já existe?

Opção 1: Vincular automaticamente (Trust Email = ON)

Se o IDP diz que o email é verificado, vincula sem perguntar. Rápido, mas arriscado.

Opção 2: Verificar por email

Envia email de confirmação ao dono da conta. Só vincula se ele confirmar. Seguro.

Opção 3: Falhar

Retorna erro "User already exists". Usuário precisa resolver manualmente (admin ou Account Console).

Resumo das camadas

Camada Pergunta Configuração
Origem De onde o usuário vem? User Federation, Identity Providers
Atributos O que sabemos sobre ele? LDAP Mappers, IDP Mappers, User Attributes
Roles O que ele pode fazer? Role assignments, Group membership, Role mappers
Token O que a aplicação vê? Protocol Mappers, Client Scopes

💡 Ponto-chave

A identidade no Keycloak é construída em camadas: origem (de onde o usuário vem) / atributos (o que sabemos sobre ele) / roles (o que ele pode fazer) / token (o que a aplicação vê). Cada camada tem seus próprios mappers e configurações. Entender esse fluxo é a chave para resolver qualquer problema de "o claim não aparece no token" ou "o atributo não está sendo mapeado".

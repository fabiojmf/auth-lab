Gerenciamento de Usuários

No Keycloak, existem duas origens fundamentais de usuários. Entender essa distinção é o primeiro passo para arquitetar qualquer solução de identidade.

👤 Usuários Locais

Criados diretamente no Keycloak

  • Armazenados no banco do Keycloak
  • Credenciais gerenciadas pelo Keycloak
  • Ideal para ambientes pequenos ou dev
  • Simples de configurar

🔗 Usuários Federados

Vêm de uma fonte externa

  • Armazenados em LDAP, AD ou outro sistema
  • Credenciais validadas na fonte externa
  • Ideal para enterprise com base existente
  • Requer configuração de federation

Criando usuários no Keycloak

Existem duas formas principais de criar usuários locais:

  1. Acesse Realm / Users / Add user
  2. Preencha username (obrigatório), email, firstName, lastName
  3. Marque Email Verified se já confirmou o email
  4. Salve e vá na aba Credentials para definir a senha
  5. Escolha se a senha é temporária (usuário troca no primeiro login) ou permanente

POST para /admin/realms/{realm}/users

{
  "username": "maria.silva",
  "email": "maria@empresa.com",
  "firstName": "Maria",
  "lastName": "Silva",
  "enabled": true,
  "emailVerified": true,
  "credentials": [{
    "type": "password",
    "value": "senha-inicial",
    "temporary": true
  }]
}

Requer token de admin com role manage-users.

Atributos do usuário

No Keycloak 20+, o recurso User Profile (agora GA) permite definir quais atributos existem, quais são obrigatórios, quem pode editá-los e validações. Configure em Realm Settings / User Profile.

Atributo Tipo Descrição
username Padrão Identificador único de login
email Padrão Email do usuário (pode ser usado como username)
firstName / lastName Padrão Nome e sobrenome
enabled Padrão Se o usuário pode logar (true/false)
cpf, matricula, etc. Custom Atributos personalizados — qualquer chave/valor

Custom attributes ficam em User / Attributes e podem ser mapeados para claims no token via Protocol Mappers.

Required Actions

Required Actions são ações que o Keycloak força o usuário a executar no próximo login. Você pode atribuí-las individualmente ou como padrão para novos usuários.

📧
Verify Email

Envia email de confirmação antes de permitir acesso

🔑
Update Password

Força troca de senha (usado com senhas temporárias)

📱
Configure OTP

Obriga configuração de segundo fator (TOTP)

📝
Update Profile

Pede que o usuário complete dados faltantes

Groups vs Roles

Ambos organizam usuários, mas com propósitos diferentes. Confundir os dois é um erro comum.

📁 Groups

Estrutura organizacional

  • Representam departamentos, equipes, unidades
  • Hierárquicos (grupos dentro de grupos)
  • Ex: /TI/Desenvolvimento, /RH/Recrutamento
  • Não concedem permissões diretamente

🛡️ Roles

Permissões e capacidades

  • Representam o que o usuário pode fazer
  • Flat (composites permitem agrupar roles, mas não formam árvore)
  • Ex: admin, viewer, editor, approver
  • Concedem acesso a funcionalidades

✅ Best Practice: Roles atribuídas a Groups

👤 Maria entra no grupo /TI/Desenvolvimento
→
📁 Grupo tem as roles developer, deploy-staging
→
🛡️ Maria herda as roles automaticamente

Quando Maria mudar de equipe, basta trocar o grupo. As roles antigas saem e as novas entram — sem editar permissões individualmente.

Habilitando e desabilitando usuários

O campo enabled controla se o usuário pode autenticar. Desabilitar um usuário:

  • Não invalida sessões ativas imediatamente (depende da expiração do token)
  • Impede novos logins e refresh de tokens
  • Preserva todos os dados do usuário (diferente de deletar)

Use para: funcionários afastados, contas suspeitas, desligamentos (antes de deletar).

Senhas temporárias vs permanentes

Tipo Comportamento Quando usar
Temporária Usuário é forçado a trocar no primeiro login Criação de conta, reset de senha pelo admin
Permanente Funciona até expirar pela política de senha Migração de sistemas, contas de serviço

💡 Ponto-chave

Usuários locais são simples e rápidos de configurar — perfeitos para desenvolvimento e ambientes pequenos. Mas em cenários enterprise, onde a empresa já tem milhares de usuários em LDAP ou Active Directory, duplicar essa base no Keycloak não faz sentido. Para isso existe a User Federation — o próximo módulo.

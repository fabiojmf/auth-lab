Identity Providers (Brokering)

Identity Brokering permite que usuários façam login no Keycloak usando outro sistema de identidade. O Keycloak não valida a senha — ele confia no resultado da autenticação feita pelo IDP externo.

Federation vs Brokering — a distinção crítica

📂 User Federation

Keycloak CONSULTA a fonte

  • Conecta a um user store (LDAP, DB)
  • Keycloak recebe username + senha
  • Keycloak pergunta ao LDAP: "senha correta?"
  • Usuário digita credenciais no Keycloak
  • Protocolo: LDAP, custom SPI

🔀 Identity Brokering

Keycloak REDIRECIONA para o IDP

  • Conecta a um identity provider (outro SSO)
  • Keycloak nunca vê a senha
  • IDP externo autentica e devolve um token/assertion
  • Usuário digita credenciais no IDP externo
  • Protocolo: OIDC, SAML

Casos de uso

🌐
Social Login

"Login com Google/GitHub" — provedores sociais pré-configurados

🏢
SSO Corporativo

"Login com minha empresa" — IDP corporativo (Azure AD, Okta, outro Keycloak)

⚖️
Multi-organização

"Login com outro Keycloak" — federação entre realms ou instâncias diferentes

Como funciona — passo a passo

👤

1. Usuário clica em "Login com IDP X" na tela de login do Keycloak

↗️

2. Keycloak redireciona o navegador para o IDP X (com parâmetros OIDC/SAML)

🔐

3. Usuário se autentica no IDP X (digita credenciais lá, não no Keycloak)

↩️

4. IDP X redireciona de volta ao Keycloak com um token/assertion assinado

🔗

5. Keycloak valida a resposta, cria/vincula um usuário local e executa o First Login Flow

🎫

6. Keycloak emite seu próprio token para a aplicação — com claims do usuário local

A aplicação nunca sabe que o usuário veio de outro IDP. Ela só vê o token do Keycloak.

Client Secret Secret do client no IDP externo ********
Mappers Transformam claims do IDP em atributos do KC email / email, name / firstName
Sync Mode Quando o mapper atualiza dados: IMPORT (só 1º login), FORCE (sempre), INHERIT (herda do IDP) FORCE

First Login Flow

Quando um usuário vindo de um IDP externo faz login pela primeira vez, o Keycloak precisa decidir o que fazer. O First Login Flow controla esse comportamento:

📝
Review Profile

Mostra formulário para o usuário revisar/completar dados antes de criar a conta

➕
Create User If Unique

Cria automaticamente um usuário local — mas só se não existir outro com mesmo email/username

🔗
Automatically Link Brokered Account

Se já existe um usuário local com o mesmo email, vincula automaticamente (requer Trust Email)

📧
Confirm Link Existing Account

Pede confirmação (email ou re-autenticação) antes de vincular a uma conta existente

OIDC vs SAML — qual protocolo usar?

Aspecto OIDC SAML 2.0
Formato JSON (JWT) XML
Configuração Simples (Discovery URL auto-configura) Complexa (metadata XML, certificados)
Uso típico Aplicações modernas, APIs, SPAs Governo, enterprise legado, federações
Tamanho do payload Compacto Verboso (XML)
Recomendação Prefira sempre que possível Use quando o IDP só suporta SAML

Hide on Login Page e kc_idp_hint

Nem sempre você quer mostrar o botão do IDP na tela de login:

Hide on Login Page = ON

O botão do IDP não aparece. Útil quando o IDP é usado apenas programaticamente.

kc_idp_hint

Parâmetro na URL de login que força redirect direto para um IDP específico, sem mostrar a tela de login do Keycloak:

https://keycloak/realms/meu-realm/protocol/openid-connect/auth?client_id=app&kc_idp_hint=tre-pi-keycloak&...

Cenário real

🏛️ "TRE-PI quer que seus usuários façam login via o Keycloak deles"

  1. No Keycloak do TRE-PI: registrar um client keycloak-broker (confidential, com redirect URI do seu KC)
  2. No seu Keycloak: Identity Providers / Add / OpenID Connect v1.0
  3. Alias: tre-pi-keycloak, Display Name: Login TRE-PI
  4. Discovery URL: https://sso.tre-pi.jus.br/realms/tre/.well-known/openid-configuration
  5. Client ID: keycloak-broker, Client Secret: (fornecido pelo TRE-PI)
  6. Configurar First Login Flow: criar usuário automaticamente + vincular por email
  7. Resultado: botão "Login TRE-PI" aparece na tela de login do seu Keycloak

💡 Ponto-chave

Identity Brokering = delegação de confiança. Você confia que outro sistema autenticou o usuário corretamente. O Keycloak nunca vê a senha — ele recebe um token/assertion assinado e, a partir dele, cria a identidade local. É assim que funciona "Login com Google", "Login com Azure AD" e qualquer integração entre IDPs.

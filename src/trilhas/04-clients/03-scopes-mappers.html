Scopes e Mappers

Scopes e mappers controlam o que vai dentro do token. Se o token não tem uma claim que você espera, a resposta quase sempre está aqui.

O que são Client Scopes

Um Client Scope é um container de configurações (mappers, roles, consent) que pode ser reutilizado entre vários clients. Pense nele como um "pacote de claims".

📦 Default Scopes

Sempre incluídos no token, sem precisar pedir explicitamente no request.

  • openid — obrigatório para OIDC
  • profile — name, family_name, etc.
  • email — email, email_verified
  • roles — realm e client roles

No Keycloak 20+, os defaults também incluem acr (Authentication Context Class) e web-origins (para CORS automático).

🎯 Optional Scopes

Só incluídos se a aplicação pedir via parâmetro scope=.

  • address — endereço do usuário
  • phone — telefone
  • offline_access — refresh token offline
  • Scopes customizados que você criar

O que são Protocol Mappers

Um Protocol Mapper transforma dados do usuário/sessão em claims dentro do token. Cada scope contém um ou mais mappers.

👤 User Attribute

Mapeia um atributo do usuário para uma claim no token.

Atributo department / claim department
🎭 User Realm Role

Inclui as roles do realm do usuário no token.

Roles / claim realm_access.roles
📌 Hardcoded Claim

Adiciona um valor fixo ao token, independente do usuário.

Claim env = "production" (sempre)
📝 User Session Note

Mapeia dados da sessão (ex: IP, método de autenticação).

Session note clientAddress / claim ip
👥 Group Membership

Inclui os grupos do usuário como claim no token. Claim típica: groups. Muito usado em ambientes corporativos.

Grupos do usuário / claim groups

Full Scope Allowed

✅ On (padrão): O token inclui todas as roles do usuário, de todos os clients e do realm.

Token contém: "realm_access": { "roles": ["admin", "user"] } "resource_access": { "outro-client": { "roles": ["manage"] } }

⚠️ Conveniente, mas o token pode ficar grande e expor roles desnecessárias.

🔒 Off: O token só inclui roles explicitamente atribuídas ao scope deste client.

Token contém: "realm_access": { "roles": ["user"] } "resource_access": { }

✅ Mais seguro. Use quando quer limitar o que cada client pode ver. Configure as roles na aba "Client Scopes" / "Scope".

Dedicated Scope (Keycloak 20+)

Cada client tem um scope dedicado ({clientId}-dedicated) visível na aba "Client Scopes". É onde ficam mappers específicos daquele client — diferente dos scopes compartilhados entre vários clients.

Se você precisa adicionar uma claim que só aparece no token de um client específico, adicione o mapper no dedicated scope dele.

Caso prático: "Meu token não tem o email!"

1 Verifique se o scope email está nos Default Scopes do client
2 Se é optional, a aplicação precisa pedir scope=openid email no request
3 Dentro do scope email, verifique se o mapper "email" existe e está configurado para incluir no access token (não só no id_token)
4 Verifique se o usuário realmente tem o atributo email preenchido

💡 Resumo

Client Scope = pacote reutilizável de mappers e configurações.
Protocol Mapper = regra que coloca dados no token (atributo / claim).
Default scope = sempre incluído. Optional = só se pedido no request.
Full Scope Allowed = On inclui tudo, Off restringe. Desabilite para segurança.
Token sem claim? / Verifique: scope atribuído? mapper existe? "Add to access token" marcado?

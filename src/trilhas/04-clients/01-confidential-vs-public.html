Confidential vs Public

Antes de configurar qualquer client no Keycloak, existe uma pergunta fundamental que define tudo:

🔑

A aplicação consegue guardar um segredo?

Se sim / Confidential. Se não / Public.

Um "segredo" aqui é o client_secret — uma credencial que o client usa para se autenticar no Keycloak ao trocar o authorization code por tokens.

Comparação visual

🔒 Confidential

Client authentication: On
Tem client_secret: Sim
Onde roda: Servidor (backend)
Código acessível ao usuário: Não
Troca de token: code + secret
Exemplos:
  • Spring Boot / Node.js / Django
  • Microserviços (service account)
  • BFF (Backend for Frontend)

🌐 Public

Client authentication: Off
Tem client_secret: Não
Onde roda: Browser ou dispositivo do usuário
Código acessível ao usuário: Sim
Troca de token: code + PKCE
Exemplos:
  • SPA (React, Angular, Vue)
  • App mobile (iOS, Android)
  • App desktop (Electron)

Cenários reais

☕ Backend web (Spring Boot)
Confidential

O código roda no servidor. O client_secret fica em variável de ambiente ou vault. Ninguém de fora tem acesso.

⚛️ SPA (React)
Public

Todo o código roda no browser do usuário. Qualquer secret no código pode ser extraído com F12. Usa PKCE para proteger a troca.

📱 App mobile (iOS/Android)
Public

O APK/IPA pode ser descompilado. Secrets embutidos no app não são seguros. Usa PKCE + Custom Tabs para login.

🤖 Microserviço chamando API
Confidential (Service Account)

Sem usuário humano. O serviço se autentica como ele mesmo usando client_credentials. O secret fica seguro no servidor.

Como a troca de token difere

O backend envia o client_secret junto com o code:

POST /realms/meu-realm/protocol/openid-connect/token grant_type=authorization_code &code=abc123 &client_id=minha-app-backend &client_secret=meu-secret-super-seguro &redirect_uri=https://app.com/callback

✅ O secret prova que é realmente a aplicação fazendo a troca, não um atacante que interceptou o code.

O SPA/mobile envia o code_verifier (PKCE) em vez do secret:

POST /realms/meu-realm/protocol/openid-connect/token grant_type=authorization_code &code=abc123 &client_id=minha-spa &code_verifier=dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk &redirect_uri=http://localhost:3000/callback

✅ O Keycloak confere se o hash do code_verifier bate com o code_challenge enviado no início. Sem secret, mas ainda seguro.

💡 Boa prática moderna

O OAuth 2.1 tornará PKCE obrigatório para todos os clients (inclusive confidential). Use secret + PKCE juntos quando possível — é defesa em profundidade.

⚠️ Histórico: versões antigas do Keycloak tinham um terceiro tipo "bearer-only" para APIs que só validam tokens. No Keycloak 20+, isso foi removido — use um client confidential com todos os flows desabilitados.

💡 Resumo

Confidential = aplicação com backend que pode guardar um secret de forma segura.
Public = aplicação onde o código é acessível ao usuário (browser, mobile). Usa PKCE.
No Keycloak: a diferença é um toggle — "Client authentication: On" = confidential, "Off" = public.
Na dúvida: se o código roda no dispositivo do usuário, é public.

Troubleshooting de Clients

Os erros mais comuns ao configurar clients no Keycloak, com explicação de por que acontecem e como resolver.

invalid_client unauthorized_client invalid_redirect_uri CORS blocked Token sem claims invalid_grant user_temporarily_disabled invalid_scope 401 da API
invalid_client
Como aparece:
HTTP 401 { "error": "invalid_client", "error_description": "Invalid client or Invalid client credentials" }
Por que acontece:
  • O client_secret está errado ou expirado
  • O client_id não existe no realm
  • O client é public mas você está enviando um secret
  • O secret foi regenerado no Keycloak mas a aplicação ainda usa o antigo
✅ Como resolver:
  1. Confira o client_id — é case-sensitive
  2. Vá em Clients → seu client → Credentials → copie o secret atualizado
  3. Se o client é public, não envie client_secret no request
  4. Verifique se está apontando para o realm correto na URL
unauthorized_client
Como aparece:
HTTP 400 { "error": "unauthorized_client", "error_description": "Client not allowed to initiate browser login with given response_type" }
Por que acontece:
  • Você está tentando usar um fluxo que não está habilitado para este client
  • Exemplo: tentando Authorization Code mas "Standard Flow" está Off
  • Exemplo: tentando client_credentials mas "Service accounts" está Off
  • Exemplo: tentando password grant mas "Direct Access Grants" está Off
✅ Como resolver:
  1. Identifique qual grant_type/response_type você está usando
  2. Vá em Clients → seu client → Settings
  3. Habilite o flow correspondente:
    • response_type=code → Standard Flow: On
    • grant_type=client_credentials → Service accounts: On
    • grant_type=password → Direct Access Grants: On
invalid_redirect_uri
Como aparece:
Página de erro do Keycloak: "Invalid parameter: redirect_uri" Ou no JSON: { "error": "invalid_redirect_uri" }
Por que acontece:
  • A redirect_uri no request não bate com nenhuma URI cadastrada no client
  • Diferença de trailing slash: /callback vs /callback/
  • Diferença de protocolo: http vs https
  • Diferença de porta: localhost:3000 vs localhost:8080
  • Wildcard usado incorretamente
✅ Como resolver:
  1. Compare EXATAMENTE a URI do request com as cadastradas no Keycloak
  2. Regras de wildcard:
    • ✅ https://app.com/* — wildcard no final do path
    • ❌ https://*.app.com/callback — wildcard no hostname NÃO funciona
    • ❌ https://app.com/*/callback — wildcard no meio NÃO funciona
  3. Em dev, use: http://localhost:3000/*
  4. Em produção, seja específico: https://app.com/auth/callback
CORS blocked
Como aparece:
Console do browser: Access to fetch at 'https://keycloak/...token' from origin 'http://localhost:3000' has been blocked by CORS policy: No 'Access-Control-Allow-Origin' header is present.
Por que acontece:
  • O campo Web Origins do client não inclui a origem da sua aplicação
  • O browser faz um preflight (OPTIONS) e o Keycloak rejeita porque não reconhece a origem
  • Só afeta chamadas do browser (SPAs). Backends não têm CORS.
✅ Como resolver:
  1. Vá em Clients → seu client → Settings → Web Origins
  2. Adicione + (sinal de mais) — isso deriva automaticamente das Valid Redirect URIs
  3. Ou adicione explicitamente: http://localhost:3000
  4. ⚠️ Não use * em produção — permite qualquer origem
  5. Lembre: Web Origins é só scheme + host + port (sem path)
Token sem roles/claims esperadas
Como aparece:
// Você decodifica o token e... { "sub": "...", "preferred_username": "maria" // cadê o email? cadê as roles? }
Por que acontece:
  • Full Scope Allowed: Off e as roles não foram atribuídas ao scope do client
  • O scope (ex: email) não está nos Default Scopes do client
  • O mapper existe mas "Add to access token" está desmarcado (só vai no id_token)
  • O usuário não tem o atributo preenchido
  • É um token de service account (não tem dados pessoais por padrão)
✅ Como resolver:
  1. Verifique Client → Client Scopes → Default/Optional scopes atribuídos
  2. Dentro do scope, verifique se o mapper tem "Add to access token: On"
  3. Se Full Scope Allowed está Off, vá em Client Scopes → Scope → atribua as roles
  4. Use o "Evaluate" do Keycloak: Client Scopes → Evaluate → veja o token gerado
invalid_grant
Como aparece:
HTTP 400 { "error": "invalid_grant", "error_description": "Code not valid" }
Por que acontece:
  • O authorization code expirou (padrão: 60 segundos)
  • O code já foi usado (é de uso único)
  • A redirect_uri na troca é diferente da usada no pedido de autorização
  • O code_verifier (PKCE) não bate com o code_challenge original
  • O refresh token expirou ou foi revogado
✅ Como resolver:
  1. Garanta que a troca acontece imediatamente após receber o code
  2. Nunca reutilize um code — se falhou, inicie o fluxo novamente
  3. Use EXATAMENTE a mesma redirect_uri no /auth e no /token
  4. Se usa PKCE, verifique que o code_verifier é o mesmo que gerou o challenge
  5. Para refresh tokens: verifique a configuração de expiração no realm (SSO Session Idle/Max)
user_temporarily_disabled
Como aparece:
Tela de login do Keycloak: "Account is temporarily disabled, contact your administrator" Ou via API: { "error": "invalid_grant", "error_description": "Account temporarily disabled" }
Por que acontece:
  • O Brute Force Protection do Keycloak foi ativado
  • Muitas tentativas de login com senha errada
  • O Brute Force Protection precisa ser habilitado manualmente (Realm Settings → Security Defenses → Brute Force Detection). Quando ativo, o threshold é configurável.
  • Pode acontecer em testes quando você erra a senha várias vezes
  • Também pode ser ativado por bots tentando senhas
✅ Como resolver:
  1. No Admin Console: Users → selecione o usuário → Credentials → desbloquear
  2. Ou espere o tempo de bloqueio passar (configurável no realm)
  3. Para ajustar: Realm Settings → Security Defenses → Brute Force Detection
  4. Em dev: considere desabilitar temporariamente ou aumentar o limite
  5. Em produção: mantenha habilitado, mas ajuste os thresholds
invalid_scope
Como aparece:
HTTP 400 { "error": "invalid_scope", "error_description": "Invalid scopes: xyz" }
Por que acontece:
  • O scope solicitado não existe no realm
  • O scope existe mas não está atribuído ao client
  • Typo no nome do scope (é case-sensitive)
✅ Como resolver:
  1. Verifique se o scope existe: Realm → Client Scopes
  2. Verifique se está atribuído ao client: Client → Client Scopes tab
  3. Confira se não há typo no nome do scope no request
401 Unauthorized (da API, não do Keycloak)
Como aparece:
A API retorna 401 mesmo com um token válido do Keycloak. O token foi emitido com sucesso, mas a API de destino rejeita.
Por que acontece:
  • Audience errado — o campo aud do token não inclui o client_id da API
  • Token expirado (exp já passou)
  • Issuer URL errado — comum atrás de reverse proxy (a API espera um iss diferente)
  • Falha na verificação de assinatura — chave pública errada ou JWKS endpoint inacessível pela API
✅ Como resolver:
  1. Decodifique o token (jwt.io) e verifique aud, iss, exp
  2. Verifique se a configuração de JWKS endpoint da API aponta para a URL correta do Keycloak
  3. Se atrás de proxy, garanta que o iss no token bate com o que a API espera
  4. Para audience: configure um mapper de Audience no client (Client Scopes → dedicated scope)

💡 Dica geral de debug

1. Leia a error_description — o Keycloak é bem descritivo.
2. Verifique os logs do Keycloak (nível DEBUG mostra detalhes da validação).
3. Use a aba "Sessions" e "Events" no Admin Console para ver o que aconteceu.
4. No Keycloak 20+: Events → Login Events mostra cada tentativa com detalhes do erro.

Para habilitar logs detalhados do Keycloak:

bin/kc.sh start-dev --log-level=org.keycloak:DEBUG

Em Docker:

docker logs keycloak-container --tail=100

Procure por linhas com WARN ou ERROR no momento do erro.

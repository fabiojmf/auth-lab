Service Accounts

Uma service account é um "usuário" que representa o próprio client. Não é uma pessoa — é a identidade da aplicação quando ela precisa agir sozinha, sem um humano logado.

Quando usar

🤖 Microserviço chamando outro microserviço
⏰ Job agendado (cron) que acessa uma API protegida
🔄 Integração entre sistemas (ERP → CRM)
📊 Pipeline de dados que precisa de autenticação

Regra simples: se não tem humano na frente da tela, provavelmente é service account.

Como configurar

1
Client authentication: On

Service accounts só funcionam com clients confidential (precisam do secret).

2
Service accounts roles: Enabled

Isso habilita o grant client_credentials e cria o usuário interno.

3
Standard Flow: Off (opcional)

Se o client é só para machine-to-machine, desabilite o Standard Flow — não precisa de redirect.

4
Atribuir roles na aba "Service Account Roles"

O token só terá as roles que você atribuir aqui. Pode ser realm roles ou client roles de outros clients.

O token: o que muda

👤 Token de usuário

{
  "sub": "f47ac10b-...",
  "name": "Maria Silva",
  "email": "maria@empresa.com",
  "preferred_username": "maria",
  "realm_access": {
    "roles": ["user"]
  }
}

🤖 Token de service account

{
  "sub": "a1b2c3d4-...",
  "clientId": "meu-servico",
  "clientHost": "10.0.1.50",
  "preferred_username":
    "service-account-meu-servico",
  "realm_access": {
    "roles": ["api-consumer"]
  }
}

Note: sem name, sem email (a menos que você configure manualmente no usuário da service account). O preferred_username segue o padrão service-account-{clientId}.

Na prática: curl

Obtendo um token via Client Credentials:

curl -X POST \ https://keycloak.empresa.com/realms/meu-realm/protocol/openid-connect/token \ -H "Content-Type: application/x-www-form-urlencoded" \ -d "grant_type=client_credentials" \ -d "client_id=meu-servico" \ -d "client_secret=meu-secret-aqui"

Resposta:

{ "access_token": "eyJhbGciOiJSUzI1NiIs...", "expires_in": 300, "token_type": "Bearer", "not-before-policy": 0, "scope": "profile email" }

⚠️ Note: não há refresh_token. Client credentials não emite refresh — quando expirar, peça outro token.

💡 Boa prática

Cache o access_token e reutilize até próximo da expiração. Não peça um token novo a cada request — isso sobrecarrega o Keycloak e adiciona latência. Renove quando faltar ~30 segundos para expirar.

Métodos de autenticação do client

Além de enviar o secret no body (client_secret_post), existem outros métodos:

  • client_secret_basic — secret no header Authorization (Base64 de client_id:secret)
  • private_key_jwt — JWT assinado com chave privada do client (mais seguro para produção)
  • client_secret_jwt — JWT assinado com o próprio client_secret

Em produção com requisitos de segurança elevados, prefira private_key_jwt — não transmite o secret pela rede.

Configurando o Audience (aud)

Quando sua service account chama uma API que valida o campo aud do token, você precisa configurar um mapper de audience:

  1. Vá no client → Client Scopes → dedicated scope
  2. Adicione um mapper tipo "Audience"
  3. Em "Included Client Audience", selecione o client_id da API de destino

Sem isso, o token terá "aud": "account" (padrão) e a API vai rejeitar com 401.

Erro comum

❌ Usar token de service account como se fosse de usuário

Se sua API espera claims como name, email, ou verifica se o sub é um usuário real, o token de service account vai falhar.

✅ Solução: APIs que aceitam service accounts devem verificar roles/permissions, não dados pessoais. Ou use um campo como clientId para identificar o chamador.

💡 Resumo

Service Account = identidade do client, sem humano envolvido.
Configuração: Client authentication On + Service accounts enabled.
Grant: client_credentials — só precisa de client_id + client_secret.
Token: sem name/email, sub = ID interno, username = service-account-{clientId}.
Roles: atribua na aba "Service Account Roles" do client.

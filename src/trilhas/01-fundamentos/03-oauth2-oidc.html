O que é OAuth2 e OIDC

São dois protocolos que trabalham juntos. Pense neles como camadas:

OpenID Connect (OIDC) Identidade — "quem é o usuário"
OAuth 2.0 Delegação de acesso — "o que o app pode fazer em nome do usuário"
HTTP / TLS Transporte

OIDC é construído em cima do OAuth2. Não são concorrentes — são complementares.

OAuth2: a analogia do manobrista

🚗

Você chega a um restaurante com manobrista.

Você quer que ele estacione seu carro, mas não quer dar acesso total a ele.

🔑

Você entrega a chave do manobrista.

Essa chave especial só liga o carro e anda. Não abre o porta-malas, não acessa o GPS.

✅

Isso é OAuth2.

Você (dono do recurso) autoriza alguém (aplicação) a acessar algo seu (dados) com permissões limitadas (scopes), sem entregar sua senha.

OAuth2 em termos técnicos

Vamos expandir a analogia. No restaurante, na verdade existem 4 personagens:

👤

Resource Owner

Você. O dono do carro. Decide se autoriza ou não.

💻

Client

O manobrista. Quer acesso ao seu carro (seus dados).

🔐

Authorization Server

O restaurante que emite a chave especial (token). No nosso mundo: o Keycloak.

📦

Resource Server

O estacionamento. Guarda o carro (seus dados) e só libera com a chave certa.

Exemplo concreto: você usa um app de agenda (Client) que precisa ler seus emails (Resource Server). O app te redireciona para o Google (Authorization Server), você autoriza, e o app recebe um token que só permite ler emails — não deletar, não enviar.

E o OIDC? Por que não basta o OAuth2?

O OAuth2 foi criado para autorização: "esse app pode acessar meus dados?". Mas ele não responde: "quem é esse usuário?". Imagine:

Um app recebe um access_token do Google. Ele sabe que pode acessar os emails. Mas de quem são esses emails? O OAuth2 puro não garante isso.

O OIDC (OpenID Connect) resolve adicionando um token extra — o id_token — que carrega a identidade do usuário:

OAuth2 puro

  • Emite access_token
  • "Esse app pode acessar recurso X"
  • Não diz quem é o usuário
  • Protocolo de autorização

OAuth2 + OIDC

  • Emite access_token + id_token
  • "Esse app pode acessar recurso X, e o usuário é o João"
  • Garante a identidade
  • Protocolo de autenticação + autorização

Na prática, quase todo mundo usa OIDC (que inclui OAuth2). Quando alguém diz "usamos OAuth2 para login", na verdade está usando OIDC.

Os 3 tokens

Quando o login acontece, o Keycloak pode emitir até 3 tokens. Cada um tem uma função:

access_token
O passaporte. Você apresenta ele para acessar APIs. Tem vida curta (5-30 minutos) porque se alguém roubar, o estrago é limitado.
id_token
O RG digital. Contém nome, email, identificador do usuário. A aplicação usa para saber quem logou. Não serve para acessar APIs.
refresh_token
O renovador. Quando o access_token expira, o refresh_token pega um novo sem o usuário precisar logar de novo. Vida longa (horas/dias).

Scopes: controlando o que vai no token

Quando a aplicação pede um token, ela declara quais scopes precisa. É como dizer: "preciso saber o nome e email do usuário, mais nada". Clique nos scopes abaixo para ver o que cada um adiciona ao token:

openid
profile
email
roles

Com esses scopes, o token conterá:

  • ✓ Identificador do usuário (sub)
  • ✓ Nome completo

O scope openid é obrigatório para OIDC — sem ele, não é OIDC, é OAuth2 puro. Os scopes profile e email também são padrão OIDC. Já o scope roles é específico do Keycloak (não faz parte do padrão). Os scopes são configurados no Keycloak pelo administrador; o desenvolvedor da aplicação escolhe quais pedir no momento do login.

💡 Resumo

OAuth2 = "esse app pode acessar seus dados?" (autorização)
OIDC = "e quem é esse usuário?" (identidade, em cima do OAuth2)
Tokens = access (passaporte), id (RG), refresh (renovador)
Scopes = o que a aplicação está pedindo
Keycloak = o servidor que orquestra tudo isso

Anatomia de um JWT

No módulo anterior você viu que o Keycloak emite tokens (access_token, id_token, refresh_token). Agora vamos abrir um e ver o que tem dentro.

Esses tokens são JWTs — JSON Web Tokens. Parecem uma string aleatória, mas têm estrutura clara.

A estrutura: 3 partes separadas por pontos

Header Algoritmo e tipo
.
Payload Os dados (claims)
.
Signature A assinatura

Header

Diz qual algoritmo foi usado para assinar o token e o tipo (sempre "JWT"). Também contém o kid — o identificador da chave usada na assinatura.

{ "alg": "RS256", "typ": "JWT", "kid": "5vrDgXBK..." }

Payload

Os dados reais — quem é o usuário, quando o token expira, quais permissões ele tem. Cada campo é uma "claim".

{ "sub": "user-123", "name": "Maria", "email": "maria@empresa.com", "exp": 1716300000 }

Signature

Garante que ninguém alterou o token. O Keycloak assina com sua chave privada. Quem recebe o token verifica com a chave pública. Se alguém mudar um byte do payload, a assinatura não bate.

RSASHA256(base64(header) + "." + base64(payload), chave_privada)

Decodificador interativo

⚠️ Importante

JWT não é criptografado — é apenas codificado em Base64. Qualquer um pode ler o conteúdo (inclusive você, agora). A assinatura garante que ninguém alterou o conteúdo, mas não esconde ele. Cuidado ao colar tokens de produção em sites externos — prefira decodificar localmente, como aqui.

Claims importantes

O payload contém "claims" — afirmações sobre o usuário e o token. As mais comuns:

sub Subject — identificador único do usuário
iss Issuer — quem emitiu o token (URL do Keycloak)
aud Audience — para quem o token foi emitido (qual client/API deve aceitá-lo)
exp Expiration — quando o token expira (timestamp Unix)
iat Issued At — quando o token foi emitido
azp Authorized Party — o client que pediu o token
realm_access Roles globais do usuário no realm
resource_access Roles do usuário por client específico
scope Scopes que foram concedidos

Por que isso importa na prática?

Quando algo dá errado (token rejeitado, permissão negada, usuário não reconhecido), a primeira coisa a fazer é decodificar o token e olhar as claims:

  • exp já passou? / Token expirado
  • iss é de outro ambiente? / Token de staging sendo usado em produção
  • resource_access não tem o role esperado? / Usuário sem permissão naquele client
  • azp é outro client? / Token emitido para aplicação errada

Decodificar ≠ Validar

Decodificar um JWT (como fizemos acima) é apenas ler o conteúdo. Qualquer um pode fazer isso. Validar é confirmar que o token é legítimo. São coisas diferentes:

Decodificar

  • Ler header e payload
  • Qualquer um pode fazer
  • Não garante nada

Validar

  • Verificar assinatura (chave pública)
  • Checar exp (expirou?)
  • Checar iss (veio do meu Keycloak?)
  • Checar aud (é para minha API?)

A chave pública para validar a assinatura fica disponível no endpoint JWKS do Keycloak:

GET /realms/{realm}/protocol/openid-connect/certs

Frameworks como Spring Security e bibliotecas como jose (Node.js) fazem essa validação automaticamente. Nunca confie em um token apenas decodificando — sempre valide.

💡 Ponto-chave

Saber ler e validar um JWT é a habilidade #1 de troubleshooting em SSO. Token rejeitado? Decodifique e olhe as claims. A resposta quase sempre está ali.

CORS e Web Origins

CORS (Cross-Origin Resource Sharing) é um mecanismo de segurança do browser que bloqueia requests entre origens diferentes. Em ambientes SSO, isso é inevitável: sua SPA em app.example.com precisa falar com o Keycloak em keycloak.example.com.

🌐

Origens diferentes = CORS

https://app.example.com → https://keycloak.example.com = cross-origin (domínio diferente)

Como funciona

🌐 Browser
app.example.com
OPTIONS (preflight)
🔐 Keycloak
keycloak.example.com
Resposta SEM header CORS
🚫 O preflight (OPTIONS) chega ao Keycloak, mas a resposta não inclui o header Access-Control-Allow-Origin. O browser então BLOQUEIA o request principal (GET/POST) — ele nunca é enviado.
🌐 Browser
app.example.com
OPTIONS (preflight)
🔐 Keycloak
keycloak.example.com
Access-Control-Allow-Origin: ✓
POST /token (request real)
✅ Browser permite. Token retornado com sucesso.

Importante: o browser envia um request OPTIONS (preflight) antes do request real. Se o preflight falha, o request real nunca é enviado.

Configuração no Keycloak

A configuração de CORS fica no client: Clients → seu-client → Web Origins

Valor Significado Quando usar
https://app.example.com Permite apenas essa origem específica ✅ Produção — mais seguro
+ Deriva origens das Valid Redirect URIs ✅ Prático e seguro
* Permite QUALQUER origem. Com *, qualquer site malicioso pode fazer requests autenticados ao Keycloak a partir do browser do usuário, potencialmente exfiltrando tokens ou realizando ações não autorizadas. ⚠️ Apenas dev/teste — PERIGOSO em produção

O que o + realmente faz

O + extrai scheme + host + port de cada Valid Redirect URI configurada:

Valid Redirect URIs: https://app.example.com/callback https://admin.example.com/* http://localhost:3000/* Web Origins com "+": → https://app.example.com → https://admin.example.com → http://localhost:3000

É o atalho mais usado — mantém CORS sincronizado com os redirects automaticamente.

Endpoints afetados pelo CORS: /token, /userinfo, /logout, /certs (JWKS), e /auth (para silent refresh via iframe).

Erros comuns

❌ Wildcard no hostname
https://*.example.com ✗ NÃO FUNCIONA

Wildcard só funciona no final do path (nas Redirect URIs), não no hostname. Para Web Origins, liste cada subdomínio explicitamente.

❌ Trailing slash
https://app.example.com/ ✗ barra no final = não bate

A origem deve ser exatamente scheme://host:port sem path ou barra final.

❌ Porta diferente
http://localhost:3000 ≠ http://localhost:8080

Portas diferentes = origens diferentes. Cada porta precisa ser listada.

Troubleshooting

1
Abra DevTools → Network

Procure o request que falhou. Verifique se há um OPTIONS (preflight) com status diferente de 200.

2
Verifique o erro no Console

Mensagem típica: Access to fetch at '...' from origin '...' has been blocked by CORS policy: No 'Access-Control-Allow-Origin' header is present

3
Corrija no Keycloak

Adicione a origem exata em Web Origins do client, ou use + se a Redirect URI já está configurada.

Cenário real

🚀 "CORS bloqueado após deploy em novo subdomínio"

A equipe fez deploy da aplicação em https://staging.app.example.com. O login parou de funcionar — CORS bloqueava o request ao Keycloak. A Valid Redirect URI foi atualizada, mas Web Origins não (estavam listadas manualmente em vez de usar +).

Solução: Substituíram as origens manuais por +. Agora qualquer nova Redirect URI automaticamente libera o CORS correspondente.

💡 Key Takeaway

CORS é proteção do browser apenas. curl e Postman não têm CORS — por isso funciona no terminal mas não no browser. Use + em Web Origins para manter sincronizado com Redirect URIs. Nunca use * em produção. E lembre: wildcard no hostname não funciona.

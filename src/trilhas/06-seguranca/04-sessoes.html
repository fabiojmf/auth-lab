Gerenciamento de Sessões

Uma sessão no Keycloak é o registro server-side de que um usuário está autenticado. Entender a hierarquia de sessões e seus timeouts é fundamental para evitar comportamentos inesperados de logout e expiração.

SSO Session vs Client Session

🔐 SSO Session

Uma por login do usuário no realm. Compartilhada entre todos os clients.

📱 Client Session: app-frontend

Criada quando o usuário acessou este client

📱 Client Session: app-admin

Criada quando o usuário acessou este client

📱 Client Session: api-backend

Criada quando o usuário acessou este client

Quando o usuário faz login uma vez e acessa 3 aplicações, existe 1 SSO Session e 3 Client Sessions.

Timeouts — a hierarquia de duração

Configuração em: Realm Settings / Sessions e Realm Settings / Tokens

SSO Session Max (10h default)
SSO Session Idle (30min default)
Refresh Token (= SSO Session restante) — O refresh token não tem um 'lifespan' próprio configurável — ele expira quando a SSO Session expira (idle ou max). A hierarquia real é: SSO Session Max ≥ SSO Session Idle > Access Token Lifespan.
Access Token Lifespan (5min default)

↑ Maior duração no topo, menor embaixo. O access token é o mais curto.

Timeout O que controla Default
SSO Session Idle Tempo sem atividade antes da sessão SSO expirar 30 min
SSO Session Max Tempo máximo total da sessão SSO (independente de atividade) 10 horas
Client Session Idle Idle timeout por client (se diferente do SSO) Herda do SSO
Client Session Max Max lifetime por client Herda do SSO
Access Token Lifespan Validade do access_token 5 min
Refresh Token Max Reuse Quantas vezes um refresh token pode ser reutilizado 0 (uso único)

⏱️ Regra de ouro

SSO Session Max > SSO Session Idle > Refresh Token > Access Token
Se o access token expira em 5 min, o app usa o refresh token para obter um novo. Se o refresh token expirou, o usuário precisa re-autenticar (mas se a SSO session ainda está ativa, é transparente).

Tipos de Logout

👤 User-initiated

Usuário clica "Sair"
App redireciona para o endpoint de logout do Keycloak
Keycloak invalida a SSO Session

👨‍💼 Admin-initiated

Admin revoga sessão pelo console
Users / usuário / Sessions / "Sign out"
Ou Sessions menu / revogar todas

⏰ Timeout

Sessão expira naturalmente
Idle timeout ou Max lifetime atingido
Próximo request falha silenciosamente

Dica: para logout OIDC correto, a aplicação deve enviar id_token_hint e post_logout_redirect_uri ao endpoint de logout. Sem id_token_hint, o Keycloak pode pedir confirmação ao usuário antes de encerrar a sessão.

Single Logout (SLO)

Quando o usuário faz logout de uma aplicação, todas as outras devem ser notificadas. Existem dois mecanismos:

🔙 Backchannel Logout

Como: Keycloak envia POST para cada client
Configuração: "Backchannel Logout URL" no client
Vantagem: Funciona mesmo se o browser fechou
Requisito: App precisa ter endpoint para receber

🖥️ Front-Channel Logout

Como: Browser carrega iframes de logout de cada client
Configuração: "Front-Channel Logout" habilitado
Vantagem: Simples, não precisa endpoint server-side
Limitação: Depende do browser estar aberto

Cenário real

🐛 "Usuário fez logout do App A mas continua logado no App B"

O usuário clicou "Sair" no App A. O App A redirecionou para o endpoint de logout do Keycloak. A SSO Session foi invalidada. Mas o App B não foi notificado — continuou usando o access token em cache até expirar.

Causa: Backchannel Logout URL não estava configurada no client do App B.

Solução: Configurar a Backchannel Logout URL em todos os clients que precisam de SLO. O App B precisa de um endpoint (ex: /logout/callback) que invalida a sessão local ao receber a notificação.

Offline Sessions

Quando um client solicita o scope offline_access, uma Offline Session é criada. Ela sobrevive restarts do servidor e tem timeouts próprios:

  • Offline Session Idle: padrão 30 dias
  • Offline Session Max: padrão 60 dias

Comum em apps mobile que precisam manter o usuário logado por semanas sem pedir login novamente. O token emitido é um offline token (refresh token de longa duração).

💡 Key Takeaway

Entenda a hierarquia: SSO Session / Client Sessions / Tokens. A SSO Session é o "guarda-chuva" — quando ela morre, tudo abaixo morre. Configure Backchannel Logout em todos os clients para garantir Single Logout real. E lembre: access tokens curtos (5 min) + refresh tokens são mais seguros que access tokens longos.

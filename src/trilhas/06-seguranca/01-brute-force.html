Brute Force Protection

Em um ambiente SSO, o Keycloak é o ponto único de entrada para todos os sistemas. Isso significa que um atacante que consegue a senha de um usuário ganha acesso a todas as aplicações conectadas. Proteção contra força bruta não é opcional — é essencial.

🎯

Um ponto de entrada = alvo de alto valor

Se o Keycloak autentica 20 sistemas, uma senha comprometida abre 20 portas.

Como o Keycloak detecta

O Keycloak monitora tentativas de login falhadas em duas dimensões:

👤 Por Usuário

Conta quantas vezes o login falhou para um username específico
Independente do IP de origem
Protege contra ataques distribuídos (múltiplos IPs, mesmo alvo)

🌐 Por IP

Conta falhas vindas do mesmo endereço IP
Independente do username tentado
Protege contra credential stuffing (um IP, múltiplos usuários)

Configuração

Acesse: Realm Settings / Security Defenses / Brute Force Detection

⚠️

Não vem habilitado por padrão! Você precisa ativar manualmente. Muitos ambientes de produção estão desprotegidos simplesmente porque ninguém ligou essa opção.

Parâmetro Descrição Default
Enabled Liga/desliga a proteção Off
Max Login Failures Quantas tentativas antes de bloquear 30
Wait Increment Seconds Tempo de bloqueio inicial (dobra a cada falha) 60
Max Wait Seconds Tempo máximo de bloqueio (teto) 900 (15 min)
Failure Reset Time Após quanto tempo sem falhas o contador zera 43200 (12h)
Quick Login Check (ms) Se duas tentativas vêm em menos que isso, conta como ataque 1000
Permanent Lockout Se ativado, bloqueia permanentemente (precisa admin desbloquear) Off

O que acontece quando é acionado

Quando o limite é atingido, o usuário recebe o erro:

error=user_temporarily_disabled

Na tela de login, aparece uma mensagem genérica como "Invalid username or password" (por segurança, não revela que a conta está bloqueada).

Simulação interativa

👤 usuario@empresa.com Max Failures: 5 | Wait: 60s
1 Tentativa com senha errada ❌ Invalid credentials
Falhas: 1/5 🟢 Conta ativa

Como desbloquear

1
Via Admin Console

Users / selecione o usuário / menu de ações (⋮) / 'Unlock user'. Em versões mais antigas, pode estar na aba Sessions.

2
Esperar o timeout

Após o Wait Increment Seconds expirar, o usuário pode tentar novamente (a menos que Permanent Lockout esteja ativo)

Via API REST (automação):

DELETE /admin/realms/{realm}/attack-detection/brute-force/users/{userId}

Útil para scripts de suporte e automação em ambientes com muitos usuários.

No Event Log

Quando o brute force é acionado, o log de eventos mostra:

Type: LOGIN_ERROR Client: account-console User: usuario@empresa.com IP: 192.168.1.100 Error: user_temporarily_disabled Details: username=usuario@empresa.com

Cenário real

🐛 "O dev testou com a senha errada 5 vezes"

Um desenvolvedor estava testando a integração e digitou a senha errada repetidamente. O usuário ficou bloqueado. Ninguém sabia por quê — a mensagem de erro era genérica. Só descobriram olhando os eventos no Admin Console.

Solução: Admin desbloqueou via "Clear user login failures". Time ajustou o Max Login Failures para 10 no ambiente de dev.

💡 Key Takeaway

Equilíbrio entre segurança e usabilidade. Um Max Login Failures muito baixo (ex: 3) causa bloqueios frequentes de usuários legítimos. Muito alto (ex: 100) não protege contra nada. Valores entre 5-10 com wait incremental são um bom ponto de partida. Em produção, sempre habilite — o default desligado é uma armadilha.

Rotação de Chaves

O Keycloak usa pares de chaves assimétricas (RSA por padrão, também suporta ECDSA e EdDSA) para assinar tokens. O access_token e id_token são JWTs assinados. Já o refresh_token, no Keycloak 20+, é um identificador opaco (não é JWT) armazenado no servidor — portanto, não é afetado diretamente pela rotação de chaves.

Na aba Keys, você verá múltiplos tipos de chave: RSA (assinatura), HMAC/AES (criptografia), e possivelmente ECDSA. Vamos focar nas chaves de assinatura RSA, que são as mais comuns.

O que são as Realm Keys

🔑
Par de chaves RSA

Chave privada (assina tokens) + chave pública (verifica assinaturas). A privada nunca sai do Keycloak.

🏷️
kid (Key ID)

Cada chave tem um identificador único. O header do JWT contém o kid que indica qual chave assinou aquele token.

📍
Onde ver

Realm Settings → Keys → aba "Active" mostra as chaves em uso.

O header do JWT referencia a chave

{ "alg": "RS256", "typ": "JWT", "kid": "abc123-def456-ghi789" ← identifica qual chave assinou }

Quando uma aplicação recebe um token, ela usa o kid para buscar a chave pública correta no endpoint JWKS e verificar a assinatura.

Bibliotecas OIDC bem implementadas (Spring Security, keycloak-js) atualizam o JWKS automaticamente quando encontram um kid desconhecido. Verifique se sua biblioteca tem 'JWKS rotation support'. Se não tiver, reiniciar a aplicação limpa o cache.

O endpoint JWKS

GET /realms/{realm}/protocol/openid-connect/certs Retorna todas as chaves públicas ativas do realm em formato JWK Set. As aplicações devem consultar este endpoint periodicamente — não cachear para sempre.

Por que rotacionar

  • Boa prática de segurança: limitar o tempo de vida de uma chave reduz o impacto de comprometimento
  • Chave comprometida: se a chave privada vazou, rotacionar imediatamente
  • Compliance: regulamentações podem exigir rotação periódica (ex: PCI-DSS)
  • Migração: mudança de algoritmo (ex: RS256 → RS512)

O que acontece quando você rotaciona

⚡
Novos tokens são assinados com a nova chave
💥
Tokens antigos ficam INVÁLIDOS se a chave antiga for removida
🔄
Aplicações com cache da chave pública antiga não conseguem verificar novos tokens até atualizar o JWKS

Active vs Passive keys

🟢 Active

Função: Assina novos tokens
No JWKS: Sim, aparece
Quantidade: Uma por algoritmo
Uso: Chave principal em uso

🟡 Passive

Função: NÃO assina novos tokens
No JWKS: Sim, ainda aparece
Quantidade: Pode ter várias
Uso: Permite verificar tokens antigos durante transição

Rotação segura — passo a passo

1
Adicionar nova chave

Gere uma nova chave e marque como Active. A chave antiga automaticamente vira Passive.

Nova (Active) — assina tokens Antiga (Passive) — ainda valida tokens antigos
2
Esperar tokens antigos expirarem

Aguarde pelo menos o Access Token Lifespan (geralmente 5 min). No Keycloak 20+ com refresh tokens opacos, apenas os access_tokens em circulação são afetados. Se usar offline tokens (JWT), aguarde o Offline Session Max.

Nova — assinando novos tokens Antiga — validando tokens em circulação
3
Remover chave antiga

Quando não há mais tokens assinados com a chave antiga em circulação, remova-a com segurança.

Nova — única chave ativa Antiga — removida ✓

Como fazer no Admin Console: Realm Settings → Keys → aba Providers → adicione um novo provider 'rsa-generated' com prioridade maior que o existente. A chave com maior prioridade se torna Active automaticamente. A chave antiga fica como Passive (ainda valida tokens antigos).

Cenário real: o incidente clássico

🔥 "Pods reiniciaram, chaves regeneraram, todos os usuários receberam 401"

O Keycloak estava rodando sem persistência de banco para as chaves (ou com configuração efêmera). Quando os pods reiniciaram, novas chaves foram geradas. Todos os tokens em circulação — assinados com as chaves antigas que não existem mais — passaram a retornar 401 Unauthorized.

Impacto: Todos os usuários de todas as aplicações foram deslogados simultaneamente.

Lição: As chaves do realm devem ser persistidas em banco de dados. Nunca use armazenamento efêmero para o Keycloak em produção.

💡 Key Takeaway

Rotação de chaves invalida tokens existentes. Planeje a rotação: mantenha a chave antiga como Passive durante a transição, espere os tokens expirarem, e só então remova. Aplicações devem buscar o JWKS periodicamente (não cachear para sempre). E nunca rode Keycloak sem persistência de chaves em produção.
